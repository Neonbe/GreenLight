# V1.0.0 — 三灯语义重设计

> **核心变更**：红灯从"系统 reject"变更为"用户 reject"，彻底消除虚假紧迫感。  
> **关联**：替代此前的 pending/blocked/dismissed/cleared 四态模型。

---

## §1 问题根因

当前三灯映射以**系统事件**为中心：

| 灯色 | 当前含义 | 问题 |
|------|---------|------|
| 🔴 | GK 拦截 (`.blocked`) | 扫描发现的旧 app 也被标红，制造虚假紧迫感 |
| 🟡 | 用户忽略 (`.dismissed`) | 只有用户操作后才会出现，黄灯区长期空置 |
| 🟢 | 已放行 (`.cleared`) | 语义正确 |

**根因**：红灯语义绑定到了"系统发现"（detection），而非"用户决策"（user action）。

---

## §2 新三灯语义

以**用户意图**为中心，重新定义：

| 灯色 | 标签 | 状态值 | 含义 | 进入条件 |
|------|------|--------|------|---------|
| 🟡 | **DETECTED** | `.detected` | 待处理 — 系统发现的全部 quarantined app | 任何来源检出（实时/扫描/主动） |
| 🔴 | **REJECTED** | `.rejected` | 用户丢弃 — 用户主动 Move to Trash | 用户点击"丢弃"按钮 |
| 🟢 | **CLEARED** | `.cleared` | 已放行 — quarantine 已移除 | 用户点击"放行"按钮 |

```swift
enum Status: String, Codable {
    case detected  // 🟡 扫描/检测发现，待用户决策
    case rejected  // 🔴 用户主动丢到垃圾箱
    case cleared   // 🟢 用户放行（quarantine 已移除）
}
```

### §2.1 对比旧模型

| 旧状态 | 新状态 | 说明 |
|--------|--------|------|
| `.pending` | ❌ 删除 | 死代码，从未被使用 |
| `.blocked` | → `.detected` | 语义拓宽：所有发现的 app 统一为"待处理" |
| `.dismissed` | ❌ 删除 | 不再需要：未处理 = 留在 🟡，无需单独状态 |
| `.cleared` | `.cleared` | 不变 |
| *(无)* | **`.rejected`** | 新增：用户主动丢到垃圾箱 |

## §3 状态机

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  任何来源检出 ──────► 🟡 detected                     │
│                        │                             │
│                        ├── 用户点"放行" ──► 🟢 cleared │
│                        │                             │
│                        └── 用户点"丢弃" ──► 🔴 rejected│
│                              (Move to Trash)         │
│                                                      │
│  🔴 占位符 ── 30天自动清理                              │
│                                                      │
│  🟢 cleared ── 更新后重新隔离 ──► 🟡 detected           │
│                                                      │
│  文件被用户手动删除（Finder） ──► 🟡 记录静默消失          │
│  文件从 Trash 恢复 ──► 下次扫描重新进入 🟡               │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### §3.1 来源不再影响初始状态

**所有来源**（`.logStream`、`.fsEvents`、`.scan`、`.proactiveScan`）统一进入 `.detected`。不再区分"实时"和"扫描"的初始状态——红灯只由**用户动作**（丢弃）触发。

### §3.2 重复检出处理

| 当前状态 | 新事件 | 结果 |
|---------|--------|------|
| 🟡 detected | 任何来源 | 保持 🟡（不重复添加，不重弹面板*） |
| 🔴 rejected | 任何来源 | 保持 🔴（文件已在 Trash，理论上不会被扫描到） |
| 🟢 cleared | 任何来源 | 移回 🟡 detected（说明 app 更新后被重新隔离） |

*面板重弹规则见 §5。

## §4 新增动作："丢弃"（Move to Trash）

### §4.1 实现

```swift
func rejectApp(_ record: AppRecord) {
    let url = URL(fileURLWithPath: record.path)
    do {
        try FileManager.default.trashItem(at: url, resultingItemURL: nil)
        // 更新状态为 rejected，保留占位符
        record.status = .rejected
        record.rejectedDate = Date()  // 新增字段，用于 30天清理
    } catch {
        GLLog.state.error("Failed to trash: \(record.appName), \(error)")
        // 失败则不改状态，留在 🟡
    }
}
```

### §4.2 权限

- `FileManager.default.trashItem()` 是 macOS 原生 API
- **不需要额外 TCC 权限**，只要当前用户对文件有写权限
- `/Applications` 中用户安装的 app：当前用户有权移动/删除
- 系统预装 app（`/System/Applications`）：无写权限，`trashItem` 会失败 → 不改状态

### §4.3 无需监控 Trash

- GreenLight 自己执行 `trashItem`，调用成功即知结果
- 不监控 `~/.Trash` 目录
- 不监控"清倒垃圾桶"操作
- 🔴 占位符 30 天后自动清理（见 §7），与 Trash 实际状态解耦

## §5 面板重弹策略

### §5.1 规则

| 条件 | 行为 |
|------|------|
| app **不在** 🟡 列表中 | 弹面板 + 添加到 🟡 |
| app **已在** 🟡 列表中，距上次弹面板 **< N 分钟** | 不弹面板 |
| app **已在** 🟡 列表中，距上次弹面板 **≥ N 分钟** | **重弹面板** |
| app 在 🔴 或 🟢 中 | 不弹面板 |

### §5.2 实现

`AppRecord` 新增 `lastPanelShownAt: Date?` 字段：

```swift
struct AppRecord {
    // ...现有字段...
    var lastPanelShownAt: Date?  // 上次弹面板的时间
    var rejectedDate: Date?       // 被丢弃的时间（用于 30天清理）
}
```

管线判断逻辑（`deduplicator.onEvent` 中）：

```swift
if let existing = appState.detectedApps.first(where: { $0.path == event.appPath.path }) {
    // 已在 🟡 列表
    let elapsed = Date().timeIntervalSince(existing.lastPanelShownAt ?? .distantPast)
    let cooldown: TimeInterval = 10 * 60  // 10 分钟（可配置）
    if elapsed < cooldown {
        GLLog.pipeline.debug("Panel cooldown: \(event.appName), \(Int(cooldown - elapsed))s remaining")
        return  // 不弹面板
    }
    // 超过冷却期 → 更新时间戳 + 重弹
    existing.lastPanelShownAt = Date()
    DetectionPanelController.shared.show(event: event)
} else {
    // 新 app → 添加到 🟡 + 弹面板
    appState.addDetectedApp(from: event)
    DetectionPanelController.shared.show(event: event)
}
```

### §5.3 可配置参数

```swift
/// 面板重弹冷却时间（秒）。同一 app 在此时间内不会重弹面板。
/// 默认 600 秒（10 分钟）。
static let panelCooldownSeconds: TimeInterval = 600
```

后续可在设置页暴露此参数（V2 考虑）。

## §6 UI 变更

### §6.1 三灯亮灭逻辑

| 灯 | 亮的条件 |
|----|---------|
| 🔴 | `rejectedApps.count > 0` |
| 🟡 | `detectedApps.count > 0` |
| 🟢 | `clearedApps.count > 0` |

### §6.2 Lane 标签

| 位置 | 旧 | 新 |
|------|-----|-----|
| 红灯区 | REJECTED (blocked) | **REJECTED** (user trashed) |
| 黄灯区 | DISMISSED | **DETECTED** |
| 绿灯区 | CLEARED | CLEARED（不变） |

### §6.3 操作按钮重设计

**检测面板（`DetectionPanelView`）**：

| 旧按钮 | 新按钮 | 说明 |
|--------|--------|------|
| 忽略 | **不做改变** | 面板关闭，app 留在 🟡 |
| 修复 | **🔓 放行** | 移除 quarantine → 🟢 |
| 修复并打开 | **▶ 放行并打开** | 移除 quarantine + 打开 → 🟢 |
| *(无)* | **🗑 丢弃** | Move to Trash → 🔴 |

> 按钮布局：`[不做改变]  [🗑 丢弃]  [🔓 放行]  [▶ 放行并打开]`  
> "不做改变" 用 secondary 样式（灰色），"丢弃" 用 destructive 样式（红色），放行用 primary（绿色）。

**ActionBubble（列表中点击 app 图标弹出）**：

🟡 detected app 的操作：
| 按钮 | 动作 |
|------|------|
| 🔓 放行 | removeQuarantine → 🟢 |
| ▶ 放行并打开 | removeQuarantine + open → 🟢 |
| 🗑 丢弃 | trashItem → 🔴 |

🔴 rejected app 的操作：
| 按钮 | 动作 |
|------|------|
| 📂 在 Finder 中显示 | 无（文件已删，此按钮灰显或隐藏） |
| *(无特殊操作)* | 占位符记录，仅展示历史 |

🟢 cleared app 的操作：保持不变（在 Finder 中显示 + 绿灯计数）。

### §6.4 Menu Bar Badge

```swift
let detectedCount = appState.detectedApps.count  // 🟡 待处理数量
if detectedCount > 0 {
    // 显示黄色数字 badge
    Label("\(detectedCount)", systemImage: "circle.fill")
        .foregroundStyle(.yellow)
} else {
    // 绿色圆点（一切正常）
    Image(systemName: "circle.fill")
        .foregroundStyle(.green)
}
```

> 注意：badge 显示的是 **🟡 待处理数量**（`detected`），不再是 🔴。因为 🔴 是已处理（用户已丢弃），不需要用户关注。

## §7 🔴 占位符生命周期

### §7.1 30 天自动清理

```swift
func cleanupExpiredRejections() {
    let cutoff = Date().addingTimeInterval(-30 * 24 * 3600)
    rejectedApps.removeAll { record in
        guard let rejectedDate = record.rejectedDate else { return false }
        return rejectedDate < cutoff
    }
}
```

触发时机：每次 App 启动 + 每次手动扫描完成后。

### §7.2 红灯区的 UI 展示

- 红灯区显示 app 名称 + 图标（图标在 reject 时快照保存，因为文件已不存在）
- 图标下方可选显示 "已丢弃 · 3 天前" 辅助文案
- 如果红灯区为空（无任何丢弃记录），显示空状态："No rejected apps"

## §8 🟡 列表的 FileExists 校验

扫描发现的 app 可能被用户在 Finder 中手动删除（不通过 GreenLight）。

### §8.1 校验策略

**渲染时轻量检查**：在 `detectedApps` getter 中过滤：

```swift
var detectedApps: [AppRecord] {
    // 实时过滤：文件不存在的记录自动排除
    blockedApps.filter { $0.status == .detected && FileManager.default.fileExists(atPath: $0.path) }
}
```

`FileManager.default.fileExists()` 是纯本地 stat 调用，单次 < 0.1ms，即使 100 个 app 也只有 10ms。

### §8.2 不存在的记录处理

- 文件不存在 → 从 🟡 列表中消失（不进入 🔴，因为用户没有通过 GreenLink 表达"拒绝"意图）
- 下次 `save()` 时顺带持久化清理结果
- 如果后续文件恢复（从 Trash 中恢复），下次扫描重新发现 → 重新进入 🟡

## §9 Persistence 旧数据兼容

| 旧 JSON 值 | 新映射 | 理由 |
|-----------|--------|------|
| `"pending"` | → `"detected"` | 语义最接近 |
| `"blocked"` | → `"detected"` | 降级为待处理（无法知道用户是否曾看过） |
| `"dismissed"` | → `"detected"` | 取消 dismissed 状态，回到待处理 |
| `"cleared"` | → `"cleared"` | 不变 |

decode 容错逻辑：

```swift
// Persistence.loadRecords() 中
let migrated = jsonString
    .replacingOccurrences(of: "\"pending\"", with: "\"detected\"")
    .replacingOccurrences(of: "\"blocked\"", with: "\"detected\"")
    .replacingOccurrences(of: "\"dismissed\"", with: "\"detected\"")
```

## §11 验收标准

| AC | 描述 |
|----|------|
| AC1 | 所有新检出的 app（无论来源）进入 🟡 DETECTED |
| AC2 | 点击"丢弃" → app 被移到垃圾桶 + 🔴 占位符出现 |
| AC3 | 点击"放行" → quarantine 移除 + 进入 🟢 |
| AC4 | 面板关闭/超时 → app 留在 🟡（不改变状态） |
| AC5 | 同一 app 10 分钟内不重弹面板，10 分钟后可重弹 |
| AC6 | Menu Bar badge 显示 🟡 待处理数（非 🔴） |
| AC7 | 🔴 占位符 30 天后自动清理 |
| AC8 | 用户在 Finder 手动删除的 app 从 🟡 列表自动消失 |
| AC9 | 旧数据（含 pending/blocked/dismissed）正确迁移 |
| AC10 | `swift build` + `swift test` 全部通过 |
