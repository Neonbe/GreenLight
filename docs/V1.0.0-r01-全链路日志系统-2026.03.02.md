## 一、核心原则

> **日志是功能的一部分，不是事后补丁。**
>
> 每个关键节点必须有日志。链路上任何一个环节静默，就等于那个环节不存在。

### 现状问题

当前代码仅有 **5 处 `print()`**，全部只覆盖错误路径：

| 文件 | 位置 | 内容 |
|:---|:---|:---|
| `LogStreamMonitor.swift:83` | 启动失败 | `[LogStreamMonitor] 启动失败` |
| `FSEventsWatcher.swift:42` | Stream 创建失败 | `Failed to create FSEventStream` |
| `NotificationManager.swift:47` | 权限请求失败 | `权限请求失败` |
| `NotificationManager.swift:104` | 通知发送失败 | `发送失败` |
| `SettingsView.swift:77` | Login Items 失败 | `Login Items 设置失败` |

**致命盲区**：正常路径上零日志。事件经过 6 个模块、多层过滤/去重，任何一层静默丢弃都无法排查。

### 事件流水总览

> 日志的使命是"尽量全程监控"——每个节点都要有日志，不怕多。功能采用（哪些事件真正触发用户通知）是另一层逻辑。

```
原始系统日志 (syspolicyd)
  │
  ▼
[LogStreamMonitor]  ──GK关键词过滤──▶ 解析 file:// 路径
  │                                    │
  │ (有路径)                            │ (无路径/private)
  │                                    │
  ▼                                    ▼
DetectionEvent                     日志记录（debug）
  │                                 ↑
  │    ┌───────────────────────────┘
  │    │   同时，FS 事件独立触发：
  │    │
  │    │   原始 FS 事件
  │    │     │
  │    │     ▼
  │    │   [FSEventsWatcher] ──.app过滤──▶ xattr检查
  │    │     │                              │
  │    │     │  ┌── 内部去重（3s）──────────┘
  │    │     │  │
  │    │     ▼  ▼
  │    │   DetectionEvent
  │    │     │
  ▼    ▼     ▼
[EventDeduplicator]  ──路径规范化+3s窗口合并──▶ GreenLightEvent
                                                │
                                                ▼
                                          [Pipeline onEvent]
                                             │        │
                                             ▼        ▼
                                        [AppState]  [Panel / Notification]
                                             │              │
                                             ▼              ▼
                                        状态变迁        用户操作
                                                         │
                                                         ▼
                                                   [QuarantineRemover]
```

**过滤层级说明**：
- **第 1 层**：LogStreamMonitor GK 关键词过滤（只保留含 `GK`/`quarantine`/`Xprotect` 的行）
- **第 2 层**：LogStreamMonitor `file://` 路径提取（无路径的 GK 行记录 debug 日志后丢弃）
- **第 3 层**：FSEventsWatcher 内部去重（3s 窗口，解决 Bundle 多文件触发问题）
- **第 4 层**：EventDeduplicator 跨通道合并（3s 窗口，同一 app 的 Channel A + B 事件合并）
- **第 5 层**：Pipeline dismissed 检查（已被用户忽略的 app 不再弹面板）

## 二、技术方案

### 2.1 日志框架选型

采用 Apple **os.Logger**（Unified Logging），而非 `print()`。

| 对比项 | `print()` | `os.Logger` |
|:---|:---|:---|
| 运行时可见 | Xcode Console / 终端 | Console.app + `log stream` + 终端 |
| 日志级别 | 无 | debug / info / notice / error / fault |
| 分类过滤 | 手工 grep | `--predicate 'subsystem == "..."'` |
| 性能 | 始终执行字符串拼接 | debug 级别在非调试时零开销 |
| 隐私 | 全部明文 | 支持 `private` / `public` 标注 |
| 生产环境 | 需手动移除 | debug 自动关闭，info 以上保留 |

### 2.2 日志定义

```swift
import os

enum GLLog {
    /// 子系统标识（= Bundle ID）
    static let subsystem = "com.greenlight.app"
    
    /// 按模块分 Category
    static let pipeline     = Logger(subsystem: subsystem, category: "pipeline")
    static let logStream    = Logger(subsystem: subsystem, category: "logStream")
    static let fsEvents     = Logger(subsystem: subsystem, category: "fsEvents")
    static let dedup        = Logger(subsystem: subsystem, category: "dedup")
    static let panel        = Logger(subsystem: subsystem, category: "panel")
    static let fix          = Logger(subsystem: subsystem, category: "fix")
    static let notification = Logger(subsystem: subsystem, category: "notification")
    static let state        = Logger(subsystem: subsystem, category: "state")
    static let onboarding   = Logger(subsystem: subsystem, category: "onboarding")
}
```

### 2.3 日志级别使用规范

| 级别 | 用途 | 示例 |
|:---|:---|:---|
| `.debug` | 高频事件、原始数据（生产环境自动关闭） | 每行 log stream 原文 |
| `.info` | 关键节点正常通过 | 事件进入去重器、面板弹出 |
| `.notice` | 重要状态变更（默认持久化） | 管线启动完成、修复成功 |
| `.error` | 可恢复错误 | 权限被拒、修复失败 |
| `.fault` | 不可恢复错误 | 子进程崩溃、关键组件初始化失败 |

### 2.4 查看方式

```bash
# 终端实时查看全部 GreenLight 日志
log stream --predicate 'subsystem == "com.greenlight.app"' --style compact

# 只看管线流转
log stream --predicate 'subsystem == "com.greenlight.app" AND category == "pipeline"'

# 只看去重决策
log stream --predicate 'subsystem == "com.greenlight.app" AND category == "dedup"'
```

## 三、全链路日志节点

### 3.1 管线启动（`GreenLightApp.setupPipeline`）

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 管线初始化完成 | `.notice` | `"Pipeline started: logStream=\(logMonitor.isRunning), fsEvents=\(fsWatcher.isRunning)"` |
| 监控目录列表 | `.info` | `"Monitoring directories: \(dirs.map(\.path))"` |

### 3.2 Channel A: LogStreamMonitor

**原始行处理**：

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 进程启动成功 | `.notice` | `"log stream process started, pid=\(proc.processIdentifier)"` |
| 收到非 GK 行（无关键词） | — | 静默丢弃，不记录 |
| 收到 GK 行（含关键词） | `.debug` | `"GK line: \(line)"` |

**解析结果分类**（基于实验 A 真实日志，GK 行有多种子类型）：

| 节点 | 级别 | 格式 | 说明 |
|:---|:---|:---|:---|
| 解析成功 → 提取到 file:// 路径 | `.info` | `"Detected: \(appURL.lastPathComponent), bundleId=\(bundleId ?? "nil")"` | 主要信号 |
| GK 行含 `Prompt shown` | `.info` | `"GK prompt shown, bundleId=\(bundleId ?? "unknown")"` | 弹窗确认信号，即使未提取到路径也证明 GK 确实拦截了 |
| GK 行含 `performScan` | `.debug` | `"GK scan initiated: \(line.prefix(120))"` | 扫描开始信号 |
| GK 行含 `scan complete` | `.debug` | `"GK scan completed: \(line.prefix(120))"` | 扫描完成信号 |
| GK 行含 `evaluateScanResult` | `.debug` | `"GK evaluate: \(line.prefix(120))"` | 扫描评估结果 |
| GK 行含 `<private>` 且无 file:// | `.debug` | `"GK line contains <private>, cannot extract path: \(line.prefix(80))"` | Apple 隐私遮蔽 |
| GK 行无法匹配任何已知模式 | `.debug` | `"GK line unrecognized: \(line.prefix(120))"` | 兜底日志 |

**生命周期与看门狗**：

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 进程启动失败 | `.fault` | `"Failed to start log stream: \(error)"` |
| 进程异常退出 → 看门狗重启 | `.error` | `"log stream terminated (status=\(status)), retry #\(retryCount) in \(delay)s"` |
| 看门狗重启成功 | `.notice` | `"log stream restarted after retry #\(retryCount), new pid=\(proc.processIdentifier)"` |
| 心跳（每 60s） | `.debug` | `"logStream heartbeat: alive \(uptime)s, lines=\(totalLines), gkHits=\(gkHitCount), detections=\(detectionCount)"` |

### 3.3 Channel B: FSEventsWatcher

> **去重说明**：FSEventsWatcher 有一层内部去重（3s 窗口），用于解决 `.app` Bundle 内多文件同时触发的问题（实验 B 中同一 app 触发了 7 次）。这和 §3.4 EventDeduplicator 的跨通道去重是**独立的两层**。

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| Stream 启动成功 | `.notice` | `"FSEvents started, directories=\(dirs.count), paths=\(dirs.map(\.path))"` |
| 收到 FS 事件（非 .app） | — | 静默丢弃，不记录 |
| 收到 FS 事件（含 .app） | `.debug` | `"FS event: \(path), flags=\(flags)"` |
| .app 路径提取失败 | `.debug` | `"FS event: failed to extract .app path from: \(path)"` |
| 内部去重跳过（3s 窗口） | `.debug` | `"FSEvents dedup: skipped \(appPath) (\(elapsed)s < 3s window)"` |
| quarantine 检查：无隔离 | `.debug` | `"Quarantine check: \(appPath), hasQuarantine=false, skipped"` |
| quarantine 检查：有隔离 → 输出事件 | `.info` | `"FSEvents detected: \(appURL.lastPathComponent), bundleId=\(bundleId ?? "nil")"` |
| Stream 创建失败 | `.fault` | `"Failed to create FSEventStream for paths=\(dirs.map(\.path))"` |
| Stream 停止 | `.info` | `"FSEvents stopped"` |

**快速扫描**（`scanApps`）：

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 扫描开始 | `.info` | `"Scan started, directories=\(dirs.count)"` |
| 扫描发现隔离 app | `.info` | `"Scan found: \(fileURL.lastPathComponent)"` |
| 扫描完成 | `.notice` | `"Scan completed: \(results.count) quarantined apps found"` |

### 3.4 EventDeduplicator

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 新路径 → 开启窗口 | `.info` | `"Dedup new: \(key), source=\(event.source), window=\(windowDuration)s"` |
| 同路径 → 合并来源 | `.info` | `"Dedup merge: \(key), added source=\(event.source), total=\(sources.count)"` |
| 窗口到期 → flush | `.info` | `"Dedup flush: \(appName), sources=\(sources), bundleId=\(bundleId ?? "nil")"` |

### 3.5 管线 onEvent 回调

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 收到事件 | `.info` | `"Pipeline received: \(event.appName), sources=\(event.sources), latency=\(latencyMs)ms"` |
| 已存在于 blockedApps（addBlockedApp 跳过） | `.debug` | `"Already in blockedApps: \(event.appPath.path)"` |
| dismissed 检查 → 跳过 | `.info` | `"Skipped dismissed app: \(event.appName)"` |
| 准备弹出面板 | `.info` | `"Showing panel for: \(event.appName)"` |
| 同时发送系统通知 | `.info` | `"Sending system notification for: \(event.appName)"` |

> **`latency`** = `Date() - event.timestamp`，即从检测到（Channel A/B 产生 DetectionEvent 的时刻）到管线收到的端到端延迟。实验 A 数据基线约 1.3s。

### 3.6 DetectionPanelController

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 创建新面板 | `.notice` | `"Panel created for: \(event.appName)"` |
| 更新面板内容（多事件切换） | `.info` | `"Panel updated: \(event.appName)"` |
| 面板定位 | `.debug` | `"Panel positioned at: (\(x), \(y))"` |
| 用户点击忽略 | `.info` | `"User dismissed: \(event.appName)"` |
| 用户点击修复 | `.info` | `"User fix: \(event.appName), shouldOpen=\(shouldOpen)"` |
| 自动消失（超时） | `.info` | `"Panel auto-dismissed (timeout)"` |
| ESC 键关闭 | `.info` | `"Panel dismissed via ESC"` |
| 面板关闭完成 | `.debug` | `"Panel closed"` |

### 3.7 QuarantineRemover（含 ShellExecutor）

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 开始修复 | `.info` | `"Fix started: \(appPath.path)"` |
| 无 quarantine → 直接成功 | `.info` | `"No quarantine, already clean: \(appPath.lastPathComponent)"` |
| xattr 执行完毕 | `.debug` | `"xattr -rd executed for: \(appPath.lastPathComponent), exitCode=\(exitCode)"` |
| 验证通过 → 成功 | `.notice` | `"Fix success: \(appPath.lastPathComponent)"` |
| 验证失败 → 需要管理员权限 | `.error` | `"Fix failed (needs admin): \(appPath.lastPathComponent), exitCode=\(exitCode)"` |
| 文件不存在 | `.error` | `"Fix failed (not found): \(appPath.path)"` |
| 自动打开 app | `.info` | `"Opening app: \(appPath.lastPathComponent)"` |
| 自动打开失败 | `.error` | `"Failed to open app: \(appPath.lastPathComponent)"` |

### 3.8 AppState 状态变迁

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 新增被拦截 app | `.notice` | `"Blocked: \(event.appName), total=\(blockedApps.count)"` |
| 从已放行移回被拦截（更新后重新隔离） | `.notice` | `"Re-blocked (update): \(record.appName)"` |
| 已存在 → 跳过 | `.debug` | `"Already blocked, skip: \(event.appPath.path)"` |
| 标记为已放行 | `.notice` | `"Cleared: \(record.appName), greenLights=\(record.greenLightCount + 1), total=\(totalGreenLights + 1)"` |
| 标记为已忽略 | `.info` | `"Dismissed: \(record.appName)"` |

### 3.9 Onboarding

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 页面切换 | `.info` | `"Onboarding page: \(currentPage)"` |
| 扫描开始 | `.info` | `"Onboarding scan started"` |
| 扫描完成 | `.notice` | `"Onboarding scan done: \(scanResults.count) found"` |
| 全部放行 | `.notice` | `"Onboarding fix all: \(appState.blockedApps.count) apps"` |
| 完成 | `.notice` | `"Onboarding completed"` |

### 3.10 NotificationManager（系统通知通道）

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 权限请求结果 | `.notice` | `"Notification permission: \(status) (granted=\(granted))"` |
| 权限被拒 | `.error` | `"Notification permission denied by user"` |
| 发送检测通知 | `.info` | `"System notification sent for: \(appName)"` |
| 发送成功通知（修复后） | `.info` | `"System notification sent: greenlight for \(appName)"` |
| 通知发送失败 | `.error` | `"System notification failed: \(error)"` |
| 权限未授权 → 跳过发送 | `.debug` | `"System notification skipped (not authorized)"` |

### 3.11 Persistence（持久化）

| 节点 | 级别 | 格式 |
|:---|:---|:---|
| 存储成功 | `.debug` | `"Saved \(records.count) records, totalGreenLights=\(total)"` |

| 加载完成 | `.debug` | `"Loaded \(records.count) records (blocked=\(blocked), cleared=\(cleared))"` |
