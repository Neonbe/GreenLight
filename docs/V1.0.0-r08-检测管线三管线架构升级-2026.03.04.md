## 一、问题定义

### 1.1 当前覆盖现况

| 场景 | 当前状态 | 缺口 |
|------|---------|------|
| 启动时 N 个 unsigned app 已存在 | ✅ `initialDetectionScan` 全量检出 | — |
| 新 unsigned app 下载到 /Applications | ✅ FSEvents + 验签 | — |
| **用户双击已知 unsigned app** | ❌ **静默** | **唯一缺口** |
| 已放行 app 更新后重新 quarantine | ✅ FSEvents 触发 | — |
| 正常签名 app | ✅ 静默 | — |
| 系统 GK 批量扫描 | ✅ 静默 | — |

### 1.2 根因

初始扫描将所有 unsigned app 加入 `blockedApps` → 之后 `proactiveScan(knownPaths: blockedApps)` 将它们过滤（L3 过滤）→ 0 found → 完全静默。

用户期望：双击一个已知的 unsigned app 时，GreenLight 应弹出面板提供一键修复帮助。

### 1.3 可利用的信号

r07 实验（2026-03-04）首次揭示了 syspolicyd `evaluateScanResult` 的完整内容：

```
evaluateScanResult: <result_code>, PST: (path: <hash>), (team: <team_id>),
  (id: <code_id>), (bundle_id: <bundle_id>), <7个参数>
```

关键字段：
- **`result_code = 0`**：GK 拦截（r07 中全部 blocked app 均为 0）
- **`result_code = 2`**：GK 通过
- **`bundle_id`**：明文，r07 中所有 blocked app 均有值

> [!WARNING]
> **置信度说明**：以上结论基于 r07 单次实验（1 台 Mac mini、7 个 app、1 个 macOS 版本）。`evaluateScanResult` 是 syspolicyd 内部日志，**非 Apple 官方公开接口**，含义可能随 macOS 版本变化。方案按「高价值但非强契约」信号处理，含解析失败降级路径。

## 二、关键实验验证（已完成）

### 2.1 r07 实验结论（详见归档报告）

- 所有被 GK 拦截的 app（result_code=0）均有 `bundle_id` ✅
- evaluateScanResult 比 FSEvents 早 ~5-10s 到达 ✅
- syspolicyd 中 `path` 字段是哈希值，无法定位文件系统 ✅

### 2.2 spctl 回声实验（r09 验证，2026-03-04 18:58）

> **问题**：我们自己的 `spctl --assess` 调用是否会产生 `evaluateScanResult` 日志？  
> 如果会，则 evaluateScanResult 不能直接用于检测用户双击（否则形成自触发回声循环）。

**实验步骤**：
1. 启动 GreenLight，等待初始扫描完成
2. 在另一终端手动执行 `spctl --assess` 两次
3. 观察 GreenLight 日志中是否出现 GK 事件

**实验结果**：
```
# 手动执行 spctl
spctl --assess --type exec "/Applications/TestUnsigned.app"  → exit: 1
spctl --assess --type exec "/Applications/TestAdHoc.app"     → exit: 3

# GreenLight 日志（执行后 60s）
heartbeat: alive 120s, lines=1980, gkHits=0  ← 零命中
```

**结论**：**`spctl --assess` 不产生 syspolicyd 日志 → 回声问题不存在 → evaluateScanResult 可安全用于检测用户双击。**

## 三、方案设计

### 3.1 核心逻辑（一句话）

当 LogStreamMonitor 捕获到 `evaluateScanResult: 0`（GK 拦截）且 `bundle_id` 匹配已知 `blockedApps` → 弹浮动面板。

### 3.2 数据流

```
syspolicyd: evaluateScanResult: 0, PST: ... (bundle_id: com.test.unsigned), ...
     │
     ▼
LogStreamMonitor.parseLogLine()
     │
     ├─→ 新增解析：提取 result_code + bundle_id
     │
     ├─→ result_code ≠ 0 → 忽略（不触发 onGKActivity）← 优化点
     │
     └─→ result_code == 0（GK 拦截）
           │
           ▼
     onGKBlocked(bundleId: "com.test.unsigned")  ← 新增回调
           │
           ▼
     GreenLightApp.swift: 匹配
           │
           ├─→ blockedApps 中有此 bundleId
           │     │
           │     ├─→ 60s 内已弹过 → 跳过（冷却去重）
           │     │
           │     └─→ 弹浮动面板（复用现有 show(event:)）
           │
           └─→ blockedApps 中无此 bundleId
                 → 触发 onGKActivity()（现有 proactiveScan 流程）
```

### 3.3 正则解析（基于 r07 真实日志格式）

```swift
// 实际日志格式（r07 实证）：
// evaluateScanResult: 0, PST: (path: 3e00aa4), (team: (null)), 
//   (id: TestUnsigned), (bundle_id: com.test.unsigned), 1, 0, 1, 0, 8, 0, 0

static let evalResultPattern = try! NSRegularExpression(
    pattern: #"evaluateScanResult:\s*(\d+).*?bundle_id:\s*([^,\s\)]+)"#
)
```

> [!NOTE]
> 现有测试文件 `GreenLightTests.swift:622` 使用的格式（`granted(bundle_id: ...)`）与真实日志格式**不一致**，需一并修正。

### 3.4 与现有 onGKActivity 的关系

| 场景 | evaluateScanResult 处理 | onGKActivity | 说明 |
|------|------------------------|--------------|------|
| result=0 + bundleId 匹配 | ✅ 弹面板 | ❌ 不触发 | 已知 app，直接处理 |
| result=0 + bundleId 未匹配 | 记录日志 | ✅ 触发 | 可能是新 app，走现有流程 |
| result=0 + bundleId 为 null | 记录日志 | ✅ 触发 | 降级走现有流程 |
| result=2（通过） | 忽略 | ❌ 不触发 | **优化：省掉无意义的 proactiveScan** |
| 解析失败 | 忽略 | ✅ 触发 | 降级走现有流程，功能不退化 |

### 3.5 冷却去重

- **bundleId 维度**：同一 bundleId **60s 内只弹 1 次**面板
- 实现：`Dictionary<String, Date>` 记录上次弹面板时间

### 3.6 面板构造

Pipeline 2 需要从 `blockedApps` 构造 `GreenLightEvent` 给 `DetectionPanelController.show(event:)`：

```swift
if let matched = appState.blockedApps.first(where: { $0.bundleId == bundleId }) {
    let event = GreenLightEvent(
        appPath: URL(fileURLWithPath: matched.path),
        appName: matched.appName,
        bundleId: matched.bundleId,
        sources: [.logStream],
        timestamp: Date()
    )
    DetectionPanelController.shared.show(event: event)
}
```

> [!IMPORTANT]
> 这里直接调用 `show(event:)` 而非经过 EventDeduplicator。原因：该事件来自已知 `blockedApps`，不需要去重（已在 60s bundleId 冷却中处理），且走 Deduplicator 会额外延迟 1s 窗口。但需确保面板弹出后，FSEvents 后续到达的同一 app 事件不会二次弹面板——现有 `Already detected, skip panel`（GreenLightApp L158-168）已覆盖此场景。

## 五、风险与降级

| 风险 | 严重性 | 降级策略 |
|------|--------|---------|
| macOS 版本更新改变 evaluateScanResult 格式 | 中 | 正则解析失败 → 降级 onGKActivity，功能不退化 |
| result_code 含义反转 | 低 | 日志监控 |
| **bundle_id=null 的 blocked app** | **实际发生** | **方案 A 反向关联：FSEvents + 30s GK 时间窗口→弹面板** |
| 同 bundleId 多路径副本 | 极低 | `first(where:)` 取第一个匹配 |

## 七、延迟预期

| 场景 | 改动前 | 改动后 | 说明 |
|------|--------|--------|------|
| 已知 app 双击 → 面板 | ∞（不弹） | **~5-11s** | 从无到有 |
| 新 app → 面板 | ~2-3s | 不变 | Pipeline 1 不受影响 |
| 正常 app → 静默 | 触发 proactiveScan | **完全静默** | result=2 不再触发扫描 |
| 启动初始扫描 | ~2-3s | 不变 | — |

> [!NOTE]
> ~5-11s 延迟来自 macOS 内核处理时间（USER_MARK → evaluateScanResult 的 r07 实测数据），GreenLight 自身处理 <100ms。这不是本次方案能优化的（受限于 macOS 内部调度），但 **从「不弹」变成「会弹」** 是核心价值。
