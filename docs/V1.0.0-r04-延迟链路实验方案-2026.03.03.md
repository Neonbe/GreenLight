# GreenLight V1.0.0 — 延迟链路实验方案

> **版本**：V1.0.0-r04
> **日期**：2026-03-03
> **基于**：V1.0.0-r03 检测管线延迟优化
> **状态**：待实施

---

## 一、背景

r03 延迟优化完成后实测发现：

| 阶段 | 文档预测 | 实际观测 |
|------|---------|---------|
| FSEvents 通知 | ~0.5s | **4.57s** |
| SecStaticCode | 0.1-0.5s | 0.07s ✅ |
| Dedup 窗口 | 1s | 1.07s ✅ |
| 面板弹出 | ~0.1s | **0.79s** |
| **总计** | **~1.5s** | **~6.5s** |

**核心问题**：4.57s 的 FSEvents 延迟来自 macOS Gatekeeper 内核处理时间，非 FSEvents 投递延迟。

**待验证假设**：双击 blocked app 时，GK 日志活动是否有可识别的**时间/类型模式**，使我们能在 FSEvents 到达前就做出预判？

**当前数据不足**：
1. 仅 1 次 blocked app 点击数据
2. 用户的双击时间点未知（只能推断）
3. 日志中 GK 事件未标注与 FSEvents 的关联

---

## 二、实验目标

回答两个问题：

1. **模式问题**：双击 blocked app 产生的 GK 事件序列，是否与打开 normal app 的 GK 事件序列有**可区分的差异**？（类型、数量、密度、间隔）
2. **可用性问题**：如果存在差异，**信噪比**是否足够用作早期信号？（即误伤率可接受）

## 三、需要的能力（代码改动）

### 3.1 用户操作打点器

**问题**：现在无法知道用户何时双击了 App。

**方案**：在 App 中添加一个**键盘快捷键打点**功能（仅 DEBUG 构建有效）：

- **触发方式**：`Cmd+Shift+T`（Timestamp）
- **效果**：在日志中输出一行精确时间戳标记
- **日志格式**：`[pipeline] ⏱ USER_MARK: <timestamp_ms>`

**改动文件**：`GreenLightApp.swift`

### 3.2 GK 事件结构化时间线

**问题**：现有 GK 日志是散落的 debug 行，难以做模式分析。

**方案**：在 LogStreamMonitor 中添加一个轻量级**事件记录器**：

```
每个 GK 事件记录:
- timestamp（ms 精度）
- category（scan / evaluate / prompt / unrecognized）
- interval（距上一个 GK 事件的间隔 ms）
```

在 heartbeat 中汇总输出最近 60s 的 GK 事件时间线（仅当有事件时输出）。

**改动文件**：`LogStreamMonitor.swift`

### 3.3 FSEvents 检出关联标记

**问题**：GK 事件和 FSEvents 检出之间没有显式关联。

**方案**：在 FSEventsWatcher 检出 blocked app 时，回查最近 10s 内的 GK 事件，标注关联：

```
[fsEvents] FS detected: <appName>, correlated GK events in last 10s: 3 (scan×1, evaluate×2)
```

**改动文件**：`FSEventsWatcher.swift` + `LogStreamMonitor.swift`（暴露 `recentGKEvents`）

## 四、实验操作流程

### 准备

1. 运行 `chmod +x create_test_apps.sh && sudo ./create_test_apps.sh` 创建 4 种测试 App
2. 构建并启动 App（`./run.sh`）
3. 等待 60s 让系统安静

### 测试标的

| 类型 | App 名 | 签名状态 | Quarantine | 预期 SecStaticCode | 预期 spctl |
|------|--------|---------|:---:|---------|---------|
| A. 未签名 | TestUnsigned.app | 无签名 | ✅ | ❌ CREATE/VALIDITY 失败 → 秒判 | — |
| B. Ad-hoc | TestAdHoc.app | ad-hoc (`-`) | ✅ | ✅ 通过 | ❌ rejected |
| C. 篡改签名 | TestDamaged.app | 签名后破坏 | ✅ | ❌ VALIDITY 失败 → 秒判 | — |
| D. 对照组 | TestNormal.app | ad-hoc | ❌ | — | — |
| E. 你的 App | (现有测试 App) | ? | ✅ | 实验确认 | ❌ rejected |

### 实验步骤

| 步骤 | 操作 | 间隔 |
|------|------|------|
| 1 | `Cmd+Shift+T` → 双击 **TestUnsigned**（Type A） | 30s |
| 2 | `Cmd+Shift+T` → 双击 **TestAdHoc**（Type B） | 30s |
| 3 | `Cmd+Shift+T` → 双击 **TestDamaged**（Type C） | 30s |
| 4 | `Cmd+Shift+T` → 双击 **TestNormal**（Type D 对照） | 30s |
| 5 | `Cmd+Shift+T` → 双击你的 blocked App（Type E） | 30s |
| 6 | `Cmd+Shift+T` → 打开 Calculator | 30s |
| 7 | `Cmd+Shift+T` → 打开 Safari | — |

### 数据采集

实验结束后从日志中提取：

```
USER_MARK → 最近的 GK 事件序列 → 是否有 FSEvents 检出 → 延迟
```
