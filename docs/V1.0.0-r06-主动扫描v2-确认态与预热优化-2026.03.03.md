# GreenLight V1.0.0 — 主动扫描 v2：确认态面板与预热优化

> **版本**：V1.0.0-r06
> **日期**：2026-03-03
> **基于**：V1.0.0-r05 主动扫描实测暴露的问题
> **状态**：✅ 已实现（2026-03-03）

---

## §0 问题回顾

r05 实现的 Channel C 主动扫描存在两个实测暴露的问题：

### 问题 1：面板显示错误 App

proactiveScan 遍历 `/Applications` 找出 **所有** SecStaticCode 失败的 app（如 4 个），按文件系统顺序输出。用户双击 Antigravity，面板却先弹出 CCSwitch（因 CCSwitch 在目录中排序更前）。

**根因**：proactiveScan 无法区分"用户正在操作的 app"和"历史遗留的 unsigned app"。

### 问题 2：L2 预热未完成就被触发

启动预热扫描用完整 SecStaticCode（含资源验证），对 18 个 quarantined app 耗时 ~8s。用户 ~6s 内就双击 → 预热未完成 → L2 缓存空 → GK 触发的扫描仍需 ~2s。且两个扫描并行竞争 `knownSafePaths`（非线程安全）。

## §1 现有白名单（过滤链）现状

> 以下是当前代码中 **全部** 过滤机制的完整清单。**不存在** 任何硬编码排除列表（如 bundleId 白名单、路径前缀排除）。

| 层级 | 名称 | 位置 | 条件 | 复杂度 |
|------|------|------|------|--------|
| **L0** | .app 扩展名 | `handleFSEvent` / `proactiveScan` | `pathExtension == "app"` | O(1) |
| **L1** | quarantine xattr | `hasQuarantine(at:)` | `getxattr("com.apple.quarantine")` | 纳秒级 |
| **L2** | SecStaticCode 缓存 | `proactiveScan` 独有 | `knownSafePaths: Set<String>`，Key=`"path\|modDate"` | O(1) 查表 |
| **L3** | AppState 已知路径 | `proactiveScan` 传参 | `knownPaths: Set<String>` | O(1) 查表 |
| **去重** | FSEvents 内部去重 | `handleFSEvent` | `recentPaths` 3s 窗口 | O(1) 查表 |
| **Dedup** | EventDeduplicator | 管线层 | path + 1s 合并窗口 | O(1) 查表 |

**缺失**：
- 无 bundleId / 签名主体白名单（如 `com.apple.*`）
- 无硬编码路径排除（如 `/Applications/Utilities/`）
- L2 非线程安全（两个扫描并发写入 `Set`）
- L3 当前传空集（`knownPaths: []`），实质失效

## §2 设计方案

### §2.1 预热时机：Onboarding 流程内触发

**现状**：预热在 `setupPipeline()` 末尾触发，与用户操作时间无关。
**改为**：在 Onboarding **Step 1（BrandStep）`onAppear`** 时即开始后台预热。

**理由**：
- Onboarding 有 2 步（Brand → Trust），用户至少需要 3-5 秒完成引导
- 预热在 Brand 步骤进入时启动，到用户完成 Onboarding 并第一次操作 app 时，已有充足时间
- 对于非首次启动（已完成 Onboarding），在 `setupPipeline()` 中立即触发预热（不变）

**预热策略**：
- QoS：`.userInitiated`（提升优先级，确保尽快完成）
- SecStaticCode flag：`kSecCSDoNotValidateResources`（轻量版，仅用于预热填充 L2 缓存）
- 结果：静默丢弃 found 列表，仅保留 L2 缓存
- 线程安全：`knownSafePaths` 改为 `DispatchQueue` 串行保护

### §2.2 GK 触发流程：确认态面板

当 GK evaluate / evaluateScanResult 到达时：

```
GK evaluate (LogStream 检测到)
  ├─ ① Menu Bar 黄灯闪烁（扫描态）
  ├─ ② 后台 proactiveScan（L2 缓存已预热 → <30ms）
  │    ├─ found > 0 → 弹出确认态面板
  │    └─ found == 0 → 等待 FSEvents
  └─ ③ 等待 FSEvents 确认是哪个 App
       ├─ FSEvents 到达 → 面板无缝切换为最终态
       └─ 超时 5s → 用 proactiveScan 首个结果兜底
```

#### §2.2.1 Menu Bar 黄灯闪烁

**触发条件**：`onGKActivity` 回调触发
**状态**：`AppState.isScanning = true` → Menu Bar 黄灯 + 闪烁

**动画规范**（与 `PulseModifier` 一致）：

```swift
// 复用现有脉冲动画模式
.opacity(isPulsing ? 0.6 : 1.0)
.animation(.easeInOut(duration: 1.25).repeatForever(autoreverses: true))
```

**支持 `reduceMotion`**：关闭辅助功能动画时，黄灯静态常亮（不闪烁）。

**结束条件**：
  - FSEvents 确认 → 面板弹出 → 红灯
  - proactiveScan 全部清空（0 found）且 5s 内无 FSEvents → 绿灯
  - 超时 10s → 绿灯

#### §2.2.2 确认态面板（新增 `DetectionConfirmingView`）

**触发条件**：proactiveScan found > 0
**设计目标**：即时反馈 + 降低隐私焦虑

> **文案原则**（UX Writing Skill 4 标准）：
> - **Purposeful**：让用户知道"保护正在生效"
> - **Concise**：≤14 词，100% 可理解
> - **Conversational**：像朋友提醒，不像系统报错
> - **Clear**：不出现"扫描""监控""检测"等敏感词

**面板布局**（复用 `DetectionPanelView` 全部视觉 Token）：

```
┌──────────────────────────────────┐   width: 320
│                                  │
│  🟡 (脉冲)  Checking...         │   statusIndicator: amberColor + PulseModifier
│                                  │
│  ⊙ ⊙ ⊙     (呼吸点动画)         │   三点呼吸: easeInOut 0.6s, stagger 0.2s
│                                  │
│  Verifying a blocked app.        │   textSecondary, 13pt
│  This only takes a moment.       │   textMuted, 11pt
│                                  │
│  ▔▔▔▔▔▔▔▔▔▔▔▔▔▔▔ (进度条)       │   倒计时条（复用 countdownBar, 5s）
└──────────────────────────────────┘
```

**文案**：

| 元素 | 英文（首选） | 中文备选 |
|------|-------------|---------|
| 状态标签 | Checking... | 确认中... |
| 主文案 | Verifying a blocked app. | 正在确认一个被拦截的 App。 |
| 副文案 | This only takes a moment. | 稍等片刻即可确认。 |

> **文案说明**：
> - "Checking" 替代 "Scanning" — 避免隐私敏感联想（"扫描我的电脑？"）
> - "Verifying a blocked app" — 聚焦于"验证已知结果"，而非"正在搜索你的文件"
> - "This only takes a moment" — 管理预期，语气温和友好

**视觉 Token**（严格复用 `DetectionPanelView`）：

```swift
// 背景
bgColor     = Color(red: 22/255, green: 28/255, blue: 45/255).opacity(0.95)
glassBorder = Color.white.opacity(0.08)
.background(RoundedRectangle(cornerRadius: 16).fill(.ultraThinMaterial))
.shadow(color: .black.opacity(0.4), radius: 30, x: 0, y: 20)

// 文字
textPrimary   = Color(248, 250, 252)
textSecondary = textPrimary.opacity(0.45)
textMuted     = textPrimary.opacity(0.25)

// 确认态专用
amberColor = Color(red: 245/255, green: 158/255, blue: 11/255)  // 复用 Onboarding OB.amber
```

**入场动画**（严格复用 `NDAnimation`）：

```swift
// Phase 1: 面板容器（与 DetectionPanelView 完全一致）
withAnimation(NDAnimation.panelTransition) {        // .spring(0.35, 0.85)
    panelOpacity = 1; panelOffset = 0; panelScale = 1
}

// Phase 2: 内容交错入场
DispatchQueue.main.asyncAfter(deadline: .now() + 0.1) {
    withAnimation(NDAnimation.cardEnter) { statusOpacity = 1 }     // .spring(0.25, 0.6)
}
DispatchQueue.main.asyncAfter(deadline: .now() + 0.15) {
    withAnimation(NDAnimation.cardEnter) { dotsOpacity = 1 }
}
DispatchQueue.main.asyncAfter(deadline: .now() + 0.2) {
    withAnimation(NDAnimation.cardEnter) { messageOpacity = 1 }
}
```

**三点呼吸动画**：

```swift
// 三个圆点，交错呼吸（不使用旋转 spinner，风格统一为脉冲）
ForEach(0..<3) { i in
    Circle()
        .fill(amberColor.opacity(dotPulsing[i] ? 1.0 : 0.3))
        .frame(width: 6, height: 6)
        .animation(
            .easeInOut(duration: 0.6)
            .repeatForever(autoreverses: true)
            .delay(Double(i) * 0.2),                    // stagger 0.2s
            value: dotPulsing[i]
        )
}
```

**`reduceMotion` 支持**：静态显示三个常亮圆点 + "Checking..." 文字，无动画。

#### §2.2.3 最终确认 → 无缝替换

**触发条件**：FSEvents + assess 确认了具体 app（`Dedup flush`）

**替换动画**（避免闪烁，视觉连贯）：

```swift
// 确认态退场：快速收缩
withAnimation(NDAnimation.cardExit) {               // .spring(0.25, 1.0) 无弹跳
    dotsOpacity = 0; messageOpacity = 0
}

// 最终态入场：交错展开（与 DetectionPanelView.playEnterAnimation 完全一致）
DispatchQueue.main.asyncAfter(deadline: .now() + 0.15) {
    // 复用 DetectionPanelView 的入场序列
    withAnimation(NDAnimation.cardEnter) { statusOpacity = 1 }
    ...
}
```

**面板不重建**：通过 `DetectionPanelController.confirmWith(event:)` 方法，替换 contentView 而非销毁重建面板。保持面板位置、阴影、背景材质完全不变。

**状态切换**：
1. `AppState.isScanning = false` → Menu Bar 红灯
2. `AppState.addBlockedApp(from:)` 写入
3. 面板显示 app 名称、图标、操作按钮

#### §2.2.4 超时兜底

**条件**：proactiveScan found > 0 但 5s 内 FSEvents 未确认
**动作**：面板替换为 proactiveScan 的**第一个** found app（按路径排序，确保稳定）

**文案过渡**：

```
确认态:  "Verifying a blocked app."
   ↓ (5s 超时)
最终态:  "检测到拦截" + app 名称（与现有 DetectionPanelView 一致）
```

### §2.3 L2 缓存线程安全

```swift
private let l2Queue = DispatchQueue(label: "com.greenlight.l2cache")
private var _knownSafePaths: Set<String> = []

func containsSafePath(_ key: String) -> Bool {
    l2Queue.sync { _knownSafePaths.contains(key) }
}
func insertSafePath(_ key: String) {
    l2Queue.sync { _knownSafePaths.insert(key) }
}
var safeCacheCount: Int {
    l2Queue.sync { _knownSafePaths.count }
}
```

### §2.4 proactiveScan 不再直出面板

**改动**：proactiveScan 的 found 结果不再直接调用 `show(event:)` 和 `addBlockedApp()`。改为：

1. 将 found 结果暂存到 `pendingProactiveScanResults: [DetectionEvent]`（在 `onGKActivity` 闭包范围内）
2. 触发确认态面板（`showConfirming(count:)`）
3. 等待 FSEvents 出口确认后，替换为最终态面板
4. FSEvents 出口判断：如果当前有确认态面板 → 调用 `confirmWith(event:)` 替换

## §3 时序对比

### Before（r05）

```
T=0     GK evaluate
T=88ms  proactiveScan 开始
T=2.1s  proactiveScan: 4 found → 直接弹面板（可能弹错 app）
T=1.6s  FSEvents → Dedup → 正确 app（面板已被错误占用）
```

### After（r06）

```
T=0     GK evaluate → Menu Bar 黄灯闪烁
T=30ms  proactiveScan: N found（L2 预热完成 → 极快）
T=30ms  弹出确认态面板（"Checking..."）
T=1.6s  FSEvents → assess → Dedup flush → 确认 Antigravity
T=1.6s  面板无缝替换：Antigravity 名称 + 图标 + 修复按钮
```

**用户感知改善**：
- T=0: Menu Bar 即时反馈（黄灯闪烁 → "系统正在保护我"）
- T=30ms: 面板出现（"Checking..." → "很快就能确认"）
- T=1.6s: 面板更新为正确 app（准确、可操作）
