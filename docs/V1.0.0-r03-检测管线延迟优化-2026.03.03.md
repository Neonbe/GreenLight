# GreenLight V1.0.0 — 检测管线延迟优化

> **版本**：V1.0.0-r03
> **日期**：2026-03-03
> **基于**：V1.0.0-r02 检测管线修复
> **状态**：待实施

---

## 一、问题背景

r02 修复了误判问题（quarantine ≠ GK 拦截），引入 `spctl --assess` 二次确认。但带来了新问题：**用户双击被拦截 App 到面板弹出需要 ~7 秒**。

### 延迟拆解（Channel B 链路）

| 阶段 | 做什么 | 为什么要做 | 耗时 |
|------|--------|-----------|------|
| ① FSEvents 通知 | macOS 通知"有文件被动了" | 唯一能知道"用户打开了某个 App"的方式 | ~1s |
| ② spctl 检查 | 问 macOS"这个 App 有没有被拦截" | 区分"正常 App"和"真被拦截的 App" | **~3s** |
| ③ 去重窗口 | 等两个探头的信息合并 | 避免同一个 App 弹两次窗 | **3s** |
| ④ 弹窗 | 创建面板并显示 | — | ~0.1s |
| **总计** | | | **~7s** |

> 注：Channel A（LogStreamMonitor）可直接命中 dedup 不经过 spctl，但实测中对"damaged"类型拦截不响应，所有检出来自 Channel B。

## 二、方案选型

### 2.1 评估过的替代方案

| 方案 | 准确性 | 延迟 | App Store 兼容 | 结论 |
|------|--------|------|:---:|------|
| **SecStaticCode + spctl** | 高 | 0.3-3s | ✅ | **✅ 采用** |
| syspolicy_check (macOS 14+) | 最高 | >3s（更慢） | ✅ | ❌ 比 spctl 更慢 |
| Endpoint Security 框架 | 最高 | 0ms | ❌ 需专门 entitlement | ❌ 不可上 App Store |
| Santa (Google 开源) | 最高 | 0ms | ❌ 系统扩展 | ❌ 量级过重 |

### 2.2 SecStaticCode vs spctl 能力对比

| 检查项 | SecStaticCode | spctl |
|--------|:---:|:---:|
| 签名是否存在 | ✅ | ✅ |
| 签名是否被篡改 | ✅ | ✅ |
| 资源/文件是否完整 | ✅ | ✅ |
| **公证状态** (notarization) | ❌ | ✅ |
| **恶意软件黑名单** | ❌ | ✅ |
| **系统安全策略** | ❌ | ✅ |
| 速度 | **0.1-0.5s** | ~3s |

> ⚠️ **关键约束**：`SecStaticCode` 签名验证通过 ≠ GK 放行。Ad-hoc 签名、未公证 App 均可通过 SecStaticCode 但被 GK 拦截。因此 **SecStaticCode 只能用作 fast-reject（快速排除），不能用作 fast-accept**。

## 三、采用方案（共 4 项）

### 3.1 FSEvents NoDefer + latency 缩短

**改动**：`FSEventsWatcher.swift`
- latency 从 `1.0` 改为 `0.5`
- flags 添加 `kFSEventStreamCreateFlagNoDefer`

**效果**：首个事件立即送达，省 ~0.5s

---

### 3.2 去重窗口 3s→1s

**改动**：`GreenLightApp.swift`
- `EventDeduplicator(windowDuration: 3.0)` → `1.0`
- 补充：`deduplicator.onEvent` 中已 blocked 的 App 跳过重复弹窗

**效果**：省 2s

---

### 3.3 SecStaticCode 快速排除（双层架构）

**改动**：`GatekeeperAssessor.swift`

```
quarantine 预过滤 → SecStaticCode 快速检查 (0.1-0.5s)
                    │
                    ├── 签名异常 → 直接标记 rejected（大多数情况，秒判）
                    │
                    └── 签名正常 → spctl 兜底确认 (~3s，后台)
                                   ├── rejected → blocked（未公证等）
                                   └── accepted → 正常，写入缓存
```

**GreenLight 检测的 App 分类**：

| 类型 | SecStaticCode | spctl | 面板延迟 |
|------|:---:|:---:|:---:|
| 签名损坏（Antigravity Tools 类） | ❌ 失败 | ❌ rejected | **~1.5s**（秒判） |
| 未签名（野生 App） | ❌ 失败 | ❌ rejected | **~1.5s**（秒判） |
| 签名正常但未公证 | ✅ 通过 | ❌ rejected | ~4s（走 spctl 兜底） |
| 正常 App | ✅ 通过 | ✅ accepted | 无弹窗 |

---

### 3.4 结果缓存

**改动**：`GatekeeperAssessor.swift`
- Key = App 路径 + App 修改时间
- Value = 判定结果
- 存储 = 内存字典（App 重启重建）

**效果**：首次检查后，后续同 App 检查 **0ms**

## 四、组合效果

| 场景 | 优化前 | 优化后 |
|------|--------|--------|
| 首次：签名有问题的 App | ~7s | **~1.5s** |
| 首次：签名正常但未公证 | ~7s | ~4s |
| 再次打开同一 App | ~7s | **~1s**（缓存命中 + Dedup） |
| 正常 App（首次） | 白跑 3s spctl | SecCode 0.3s + spctl 3s 后台 → 缓存 |
| 正常 App（再次） | 白跑 3s spctl | **0ms**（缓存命中） |

## 六、退出码契约（冻结）

| spctl exitCode | 含义 | GreenLight 判定 |
|:-:|------|------|
| 0 | GK 通过 | `.accepted` |
| 1 | 签名损坏/资源缺失 | `.rejected` |
| 3 | Policy denied (未公证等) | `.rejected` |
| 其他 | 参数错误/启动失败 | `.unknown` |

## 七、风险评估

| 风险 | 严重性 | 缓解措施 |
|------|--------|---------|
| SecStaticCode 漏过"签名正常但未公证" | 中 | spctl 兜底层覆盖 |
| Dedup 1s 导致重复弹窗 | 低 | 已 blocked 不重复弹窗逻辑 |
| FSEvents NoDefer 增加回调频率 | 低 | latency 保持 0.5（非 0）+ 现有去重 |
