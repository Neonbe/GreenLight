# GreenLight V1.0.0 — 主动扫描加速方案

> **版本**：V1.0.0-r05
> **日期**：2026-03-03
> **基于**：V1.0.0-r04 延迟实验结论
> **状态**：已实施

---

## 一、问题

r04 实验证实：从用户双击到面板弹出约 **5-8 秒**，其中 **3-6.5 秒**是 macOS GK 内核处理时间（不可控），剩余 ~1.3 秒是管线处理时间。

当前检测流程（被动等待）：
```
用户双击 → GK 评估 3-6s → xattr 修改 → FSEvents 通知 → assess → Dedup 1s → 面板
                         ↑ 不可控                      ↑ 被动等待
```

r04 发现 GK evaluate 事件比 FSEvents **早到 3-5 秒**，但当前代码未利用这个信号。

---

## 二、方案：Channel C — 主动扫描

在 GK evaluate 到达时，**不等 FSEvents**，直接遍历监控目录查找 blocked app。

```
用户双击 → GK evaluate 事件 (~2s) → 主动扫描 /Applications → SecStaticCode → 面板
                                     ↑ 新增！主动出击           总计 ~2-3s
```

### 2.1 触发条件

将 `evaluateScanResult` 纳入 `onGKActivity` 触发范围（目前仅 `Prompt shown` 触发），加 **5 秒冷却**防止频繁扫描。

### 2.2 扫描逻辑（proactiveScan）

```
遍历监控目录中的 .app 文件
  → 白名单过滤（跳过已知安全 App） ....... O(1) 查表
  → hasQuarantine？ ...................... 纳秒级 getxattr
  → SecStaticCode 验签 .................. ~6ms
  → 如果失败 → 直接触发检测（跳过 Dedup）
```

### 2.3 白名单机制

**三层自动过滤，无需手动维护**：

| 层 | 过滤条件 | 作用 | 成本 |
|---|---------|------|------|
| L1 | `hasQuarantine == false` | 排除系统 App、App Store App、本地编译 App | 纳秒/个 |
| L2 | `knownSafePaths` 缓存 | 跳过已经 SecStaticCode 验证过且通过的 App（Key=路径+modDate） | O(1) 查表 |
| L3 | `AppState` 已知 | 跳过已在 blocked/dismissed/cleared 中的 App | O(1) 查表 |

典型 /Applications：~100 个 .app → L1 过滤 quarantine 后剩 0-5 个 → L2 过滤缓存后剩 0-1 个。

**扫描总耗时预估：<10ms**

### 2.4 直出（跳过 Dedup）

Channel C 检出的事件标记 `source: .proactiveScan`，pipeline 收到后**跳过 EventDeduplicator 直接弹面板**。

理由：主动扫描 + SecStaticCode 失败 = 100% 确定被 GK 拦截，无需等待 Channel A/B 汇合确认。

## 三、预期延迟对比

| 场景 | 当前 | 优化后 | 省时 |
|------|:---:|:---:|:---:|
| 未签名 App（你的 App） | ~5s | **~2s** | **3s** |
| 签名损坏 App | ~5s | **~2s** | **3s** |
| Ad-hoc 签名（未公证） | ~7s | ~7s（仍走 FSEvents） | 0 |

## 四、改动清单

| 文件 | 改动 | 工作量 |
|------|------|--------|
| `LogStreamMonitor.swift` | `evaluateScanResult` 也触发 `onGKActivity` | 1 行 |

## 五、验证

1. `swift build` + `swift test` 通过

3. 预期：`proactiveScan` 在 GK evaluate 后 <100ms 内检出，比 FSEvents 早 3-5 秒
