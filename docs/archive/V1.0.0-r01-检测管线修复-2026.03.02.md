> **版本**：V1.0.0-r01

> **日期**：2026-03-02
> **状态**：待评审

## 一、问题背景

### 1.1 用户反馈

用户在 Onboarding 完成后，点击 Antigravity Tools 时，面板弹出的却是 Sublime Text 的"被拦截"通知。更严重的是，**即使不做任何操作，面板仍持续弹出**，dashboard 显示 15 个 app 全部处于 BLOCKED 状态。但这些 app **全部都能正常使用**。

### 1.2 根因分析

#### 核心错误：`com.apple.quarantine` ≠ 被 Gatekeeper 拦截

当前代码（`FSEventsWatcher.hasQuarantine()`）的判定逻辑：

```swift
// 当前实现 ❌
static func hasQuarantine(at path: String) -> Bool {
    return getxattr(path, "com.apple.quarantine", nil, 0, 0, XATTR_NOFOLLOW) > 0
}
```

**这个函数检查的是"有没有 quarantine 扩展属性"，而不是"GK 是否拦截了这个 app"。**

根据 Apple 的安全机制和网络调研结果：

| 概念 | 含义 | 触发条件 | 能否正常使用 |
|------|------|---------|------------|
| **有 quarantine 属性** | "此文件来自互联网" | 任何从浏览器/邮件/AirDrop 下载的文件 | ✅ 大多数都能用 |
| **被 GK 拦截** | "此 app 未通过安全检查" | app 未签名/未公证/被 XProtect 标记 | ❌ macOS 拦截，无法打开 |

`com.apple.quarantine` 属性格式为 `flags;timestamp;agent;UUID`，其中 flags 字段包含关键信息：

| Flag 位 | 含义 |
|---------|------|
| `0x0040` | 用户已批准（已成功打开过/通过 GK 检查） |
| `0x0080` | 用户已被警告过 |

> 例如 flag = `0082` 表示"已被警告且已通过检查"的正常 app，它仍然有 quarantine 属性，但完全能正常使用。

**用户本机 15 个 app 有 quarantine 属性是完全正常的**（它们都是从网上下载安装的），GreenLight 错误地将它们全部标记为"被拦截"。

#### 衍生问题链

```
A. scanApps() 把所有 quarantined app 视为被拦截 ← 根因
   ↓
B. Fallback scan 被 syspolicyd 后台活动频繁触发（evaluateScanResult + <private> catch-all）
   ↓
C. 每次 scan 找到 15 个 quarantined app → 全部推入 pipeline
   ↓
D. dismissed 的 app 每次被重新标记为 blocked → 面板再次弹出
   ↓
E. 用户看到的：无操作也持续弹 Sublime Text
```

#### 原始设计文档中的定义 vs 实际实现

技术规格文档 `experiment_results_and_tech_spec.md` 第 189 行明确写到：

> **Channel B**（FSEventsWatcher）：只能检测"新文件出现"，不能检测"用户双击被拦截"的精确时刻

但实际的 `scanApps()` 实现偏离了这个设计意图——它扫描所有已存在的文件，而非只关注"新出现"的。

---

## 二、修改目标

1. 修正检测判定逻辑：**只有真正被 GK 拦截的 app 才应标记为 blocked**
2. 消除误报：已通过 GK 检查的正常 app 不应出现在 blocked 列表中
3. 杜绝无操作弹窗：用户无交互时不应弹出面板

## 三、技术方案

### 3.1 判定策略（三级信号）

| 优先级 | 信号来源 | 判定标准 | 置信度 | 对应场景 |
|--------|---------|---------|--------|---------|
| **P0** | Channel A (LogStreamMonitor) | syspolicyd 日志包含 `file://` 路径 + GK 关键词 | **最高** | GK 正在评估/拦截某个 app |
| **P1** | Channel B (FSEventsWatcher) 实时 | FSEvents 检测到 `.app` 新出现 + quarantine flag 检查 | **高** | 新下载的 app 落盘 |
| **P2** | Fallback scan | 全盘扫描 → `spctl --assess` 验证 | **中** | 兜底/初始扫描 |

> **核心原则**：`hasQuarantine()` 只是预过滤条件（排除没有 quarantine 的 app），**不是最终判定条件**。最终判定需要二次确认。

### 3.2 二次确认机制

引入 `assessGatekeeper(at:)` 函数，使用 macOS 原生 `spctl --assess` 命令判断 app 是否真正被 GK 拦截：

```swift
/// 使用 spctl --assess 判断 app 是否被 GK 拦截
/// - Returns: true = 被拦截（rejected），false = 正常（accepted）
static func isBlockedByGatekeeper(at path: String) -> Bool {
    let process = Process()
    process.executableURL = URL(fileURLWithPath: "/usr/sbin/spctl")
    process.arguments = ["--assess", "--type", "exec", path]
    process.standardOutput = FileHandle.nullDevice
    process.standardError = FileHandle.nullDevice
    
    do {
        try process.run()
        process.waitUntilExit()
        // exit code 0 = accepted, 3 = rejected
        return process.terminationStatus != 0
    } catch {
        return false // 无法判定时默认不拦截
    }
}
```

> **关键发现**（联网调研）：
> - `spctl --assess` 返回退出码 `0` = GK 通过（accepted）
> - `spctl --assess` 返回退出码 `3` = GK 拦截（rejected）
> - 这正是 Sentinel 用来判断 GK 状态的同一工具

### 3.3 各模块修改要点

#### 3.3.1 FSEventsWatcher — `scanApps()` 修正

**现状**：扫描所有有 quarantine 属性的 app → 全部视为被拦截

**修改为**：

```
扫描 .app 文件
  → 有 quarantine 属性？
    → 调用 spctl --assess 验证
      → rejected → 推入 pipeline（真正被拦截）
      → accepted → 跳过（正常 app）
```

**注意事项**：
- `spctl --assess` 每次调用约 50~200ms，15 个 app 约需 1~3 秒，可接受
- 应异步执行避免阻塞主线程
- 添加日志记录每个 app 的评估结果：`Scan assess: <appName>, accepted/rejected`

#### 3.3.2 FSEventsWatcher — `handleFSEvent()` 实时检测修正

**现状**：检测到 FS 变动 + 有 quarantine → 直接推入 pipeline

**修改为**：

```
FSEvents 检测到 .app 变动
  → 有 quarantine 属性？
    → 调用 spctl --assess 验证
      → rejected → 推入 pipeline
      → accepted → 跳过
```

#### 3.3.3 LogStreamMonitor — 削减 `onGKActivity` 触发范围

**现状**：以下三种情况都触发 `onGKActivity()`：
1. `evaluateScanResult` 匹配 ← syspolicyd 后台常规操作也会命中
2. `Prompt shown` 匹配 ← 仅在弹窗时触发，信号好
3. `<private>` catch-all ← 几乎所有 GK 日志都命中

**修改为**：
1. **删除** `<private>` catch-all（第 180-182 行）—— 这是噪声最大的分支
2. **保留** `Prompt shown` → 触发 `onGKActivity()`（高信号）
3. **修改** `evaluateScanResult` → **不再触发** `onGKActivity()`，仅记录日志（这是后台常规操作，不代表有拦截发生）

> **理由**：`Prompt shown` 意味着 macOS 已经弹窗提示用户，这才是真正的拦截即视信号。而 `evaluateScanResult` 只是 GK 在后台评估，平均每 1-2 分钟就触发一次，且大多数评估结果是通过的。

#### 3.3.4 Fallback scan 冷却机制

**现状**：`onGKActivity()` → 500ms 去抖 → 触发 scan

**修改为**：
- 最短冷却时间：**120 秒**（即使 `onGKActivity` 被触发，距上次 scan 不满 120 秒则跳过）
- 日志记录跳过原因：`Fallback scan skipped: cooldown (Xs remaining)`

```swift
private var lastScanTime: Date?
private let scanCooldown: TimeInterval = 120 // 120 秒冷却

logMonitor.onGKActivity = { [fsWatcher, deduplicator] in
    let now = Date()
    if let last = lastScanTime, now.timeIntervalSince(last) < scanCooldown {
        let remaining = Int(scanCooldown - now.timeIntervalSince(last))
        GLLog.pipeline.debug("Fallback scan skipped: cooldown (\(remaining)s remaining)")
        return
    }
    
    fallbackScanTimer?.cancel()
    let work = DispatchWorkItem {
        lastScanTime = Date()
        GLLog.pipeline.info("Fallback scan triggered by GK activity")
        let events = fsWatcher.scanApps()
        for event in events {
            deduplicator.receive(event)
        }
    }
    fallbackScanTimer = work
    DispatchQueue.global(qos: .userInitiated).asyncAfter(deadline: .now() + 0.5, execute: work)
}
```

#### 3.3.5 AppState — dismissed 稳定性

**现状**：dismissed 的 app 在下次 scan 时被重新标记为 blocked

**修改为**：
- 来自 **scan** 的事件 → 不做 re-block（scan 是批量操作，不代表新的拦截发生）
- 来自 **LogStreamMonitor 直接路径匹配** 或 **FSEvents 实时检测** 的事件 → 可以 re-block（这是新的拦截信号）

实现方式：`GreenLightEvent` 增加 `isScanResult: Bool` 属性，`addBlockedApp()` 据此判断：

```swift
func addBlockedApp(from event: GreenLightEvent) {
    // ... existing logic ...
    } else if let existingIndex = blockedApps.firstIndex(where: { $0.path == event.appPath.path }) {
        if blockedApps[existingIndex].status == .dismissed {
            if event.isScanResult {
                // 来自 scan → 不重置 dismissed 状态
                GLLog.state.debug("Scan result for dismissed app, skip: \(event.appName)")
                return
            }
            // 来自实时检测 → 重置为 blocked
            blockedApps[existingIndex].status = .blocked
            // ...
        }
    }
}
```

为了便于后续诊断，以下关键节点需增加日志（使用 `.public` privacy 在开发阶段暴露真实数据）：

| 位置 | 日志内容 | 级别 |

## 六、风险评估

| 风险 | 严重性 | 缓解措施 |
|------|--------|---------|
| `spctl --assess` 对某些 app 执行较慢 | 低 | 异步执行，不阻塞 UI |
| `spctl --assess` 对未签名 app 可能弹出系统对话框 | 中 | 需实验验证；如有此问题可改用读取 quarantine flag 的 `0x0040` 位作为替代方案 |
| 修改后遗漏某些真正被拦截的场景 | 中 | Channel A (LogStreamMonitor) 不受此修改影响，仍是最精准的检测通道 |

> [!IMPORTANT]
> **关于 `spctl --assess` 的潜在副作用**：需要先在真机上测试 `spctl --assess /path/to/unsigned.app` 是否会触发系统弹窗。如果会，则需要改用读取 quarantine xattr 的 flag 值来判断（检查 `0x0040` 位 = 用户已批准）。这两种方案的优劣比较：
>
> | 方案 | 优点 | 缺点 |
> |------|------|------|
> | `spctl --assess` | 最准确，Apple 官方判定 | 可能触发副作用 |
> | 读取 quarantine flag `0x0040` 位 | 无副作用，纯读取 | flag 含义非 Apple 官方文档化 |
>
> 建议先实验 `spctl --assess`，如有副作用再降级为 flag 方案。
