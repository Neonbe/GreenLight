# GreenLight V1.0.0 — 检测管线修复需求

> **版本**：V1.0.0-r02（基于评审修订）
> **日期**：2026-03-03
> **状态**：✅ 已实现（2026-03-03）
> **评审来源**：外部程序员 9 条发现 + 3 轮补充联网调研

---

## 一、问题背景

### 1.1 用户反馈

Onboarding 完成后，点击 Antigravity Tools → 面板弹出 Sublime Text 的"被拦截"通知。即使**无任何操作**，面板仍持续弹出。Dashboard 显示 15 个 app 全部 BLOCKED，但这些 app **全部能正常使用**。

### 1.2 根因

**核心错误**：`com.apple.quarantine` 属性 ≠ 被 Gatekeeper 拦截。

```swift
// 当前实现 ❌ — FSEventsWatcher.swift:66
static func hasQuarantine(at path: String) -> Bool {
    return getxattr(path, "com.apple.quarantine", nil, 0, 0, XATTR_NOFOLLOW) > 0
}
```

这个函数检查的是"文件来自互联网"，不是"GK 拦截了这个 app"。

| 概念 | 含义 | 触发条件 | 能否正常使用 |
|------|------|---------|------------|
| 有 quarantine 属性 | 此文件来自互联网 | 浏览器/邮件/AirDrop 下载 | ✅ 大多数能用 |
| 被 GK 拦截 | 未通过安全检查 | 未签名/未公证/XProtect 标记 | ❌ 无法打开 |

### 1.3 衍生问题链

```
A. scanApps() 把所有 quarantined app 视为被拦截 ← 根因
 → B. syspolicyd 后台活动频繁触发 fallback scan（evaluateScanResult + <private> catch-all）
   → C. 每次 scan 找到 15 个 quarantined app → 全部推入 pipeline
     → D. dismissed 的 app 被重新标记为 blocked → 面板再次弹出
       → E. 用户：无操作持续弹窗
```

### 1.4 设计偏离

原始技术文档（`experiment_results_and_tech_spec.md:189`）明确定义：

> Channel B（FSEventsWatcher）：只能检测"**新文件出现**"，不能检测"用户双击被拦截"的精确时刻

但 `scanApps()` 扫描所有**已存在**文件，偏离了"新文件出现"的设计意图。

---

## 二、修改目标

1. **修正判定**：只有真正被 GK 拦截的 app 才标记为 blocked
2. **消除误报**：已通过 GK 检查的正常 app 不出现在 blocked 列表
3. **杜绝无操作弹窗**：用户无交互时不弹面板
4. **打断反馈循环**：`spctl --assess` 产生的 syspolicyd 日志不触发新的 scan

---

## 三、技术方案

### 3.1 判定信号分级

| 优先级 | 信号来源 | 说明 | 置信度 |
|--------|---------|------|--------|
| **P0** | Channel A (LogStreamMonitor) | syspolicyd 日志含 `file://` 路径 + GK 关键词 | 最高 |
| **P1** | Channel B (FSEventsWatcher) 实时 | FSEvents 检测到 `.app` **新出现** + quarantine 预过滤 + `spctl` 二次确认 | 高 |
| **P2** | Fallback scan（目标化补漏） | 仅近期候选路径 + `spctl` 二次确认 | 中 |

> **核心原则**：`hasQuarantine()` 仅作预过滤（排除无 quarantine 的 app），**不是最终判定**。最终判定由 `spctl --assess` 完成。

### 3.2 GatekeeperAssessor 模块（新增）

> [!IMPORTANT]
> 复用现有 `ShellExecutor`（`GreenLight/Action/ShellExecutor.swift`），不重复造轮子。

#### 3.2.1 协议定义

```swift
/// 可注入依赖，便于测试
protocol GatekeeperAssessing {
    func assess(appPath: String) -> GKAssessResult
}

enum GKAssessResult {
    case accepted    // 退出码 0：GK 通过
    case rejected    // 退出码 3：GK 拦截
    case unknown     // 退出码 1/2/其他：无法判定
}
```

#### 3.2.2 生产实现

```swift
struct GatekeeperAssessor: GatekeeperAssessing {
    func assess(appPath: String) -> GKAssessResult {
        let result = ShellExecutor.run("/usr/sbin/spctl",
            arguments: ["--assess", "--type", "exec", appPath])
        
        switch result.exitCode {
        case 0:  return .accepted
        case 3:  return .rejected
        default: return .unknown
        }
    }
}
```

#### 3.2.3 退出码语义（来自 `man spctl`）

| 退出码 | 含义 | GreenLight 处理 |
|--------|------|----------------|
| 0 | accepted | 跳过，正常 app |
| 1 | general failure | unknown → 仅记录日志 |
| 2 | bad arguments | unknown → 仅记录日志 |
| 3 | denied by policy | **rejected → 标记 blocked** |

> [!WARNING]
> **反馈循环风险**：`spctl --assess` 调用本身会经过 syspolicyd，产生日志条目，被 LogStreamMonitor 捕获。必须通过收窄 `onGKActivity` 触发条件来打断循环（见 §3.4）。

#### 3.2.4 `0x0040` flag 方案定位

quarantine xattr 的 `0x0040` 位（"用户已批准"）来自社区观察，**非 Apple 官方文档**（`LSQuarantine.h` 仅定义字典键）。

**定位**：仅作为实验性参考信号，不纳入主判定链。如未来发现 `spctl --assess` 在特定场景失效，可作为辅助参考，但需标注实验性质。

### 3.3 DetectionEvent.Source 扩展

**现状**：`Source` 仅区分 `logStream` / `fsEvents`，无法区分"实时检测"和"批量扫描"。

**修改为**：

```swift
enum Source: Hashable {
    case logStream   // Channel A：syspolicyd 直接路径匹配（最高置信）
    case fsEvents    // Channel B：FSEvents 实时文件变动（高置信）
    case scan        // Fallback：批量扫描结果（低置信）
}
```

**去重合并规则**（EventDeduplicator）：
- 3 秒窗口内同路径事件照常合并 sources
- 状态决策按**最高置信来源**执行：
  - sources 含 `logStream` 或 `fsEvents` → 视为实时检测（可 re-block dismissed）
  - sources 仅含 `scan` → 走静默路径（不 re-block dismissed）

### 3.4 LogStreamMonitor — 收窄 `onGKActivity`

**要解决的问题**：
1. `<private>` catch-all 匹配几乎所有 GK 日志行 → 噪声最大
2. `evaluateScanResult` 每 1-2 分钟后台触发 → 且是 `spctl --assess` 自身的回声
3. 只有 `Prompt shown` 代表 macOS **真正弹窗拦截**

**修改**：

| 分支 | 现状 | 修改后 |
|------|------|--------|
| `Prompt shown` | 触发 `onGKActivity()` | **保留**（唯一的 scan 触发源） |
| `evaluateScanResult` | 触发 `onGKActivity()` | **仅记录日志**，不触发 scan |
| `<private>` catch-all（第 180-182 行） | 触发 `onGKActivity()` | **删除** |

> **理由**：`evaluateScanResult` 是 syspolicyd 后台常规操作的日志（约每 1-2 分钟一次），且 `spctl --assess` 本身也会产生该日志——如果保留触发会形成 `scan → spctl → evaluateScanResult 日志 → scan → ...` 反馈循环。`Prompt shown` 是唯一表示"macOS 真正弹窗给用户"的高信号。

### 3.5 Fallback scan — 从"全盘扫描"改为"目标化补漏"

**现状**：遍历 Downloads/Desktop/Applications 所有 `.app` → 全部检查 quarantine → 全部推入 pipeline。

**修改为**：

```
Prompt shown 触发 → 仅检查近期候选路径（30 秒内 FSEvents 活跃的 .app 路径）
  → 有 quarantine？ → spctl --assess → rejected → 推入 pipeline
```

**实现要点**：
- FSEventsWatcher 维护 `recentCandidates: [String: Date]`（近 30 秒内有 FS 活动的 `.app` 路径）
- Fallback scan 只遍历 `recentCandidates`，不做全盘枚举
- 冷却时间 120 秒（距上次 scan 不满 120 秒则跳过）

**用户手动全量扫描**：保留给 V2 Dashboard 的"安全体检"功能，允许慢，用户显式触发。

### 3.6 AppState — dismissed 稳定性

**修改**：`addBlockedApp()` 根据事件来源决定是否 re-block：

```swift
func addBlockedApp(from event: GreenLightEvent) {
    // ...
    } else if let existingIndex = blockedApps.firstIndex(where: { $0.path == event.appPath.path }) {
        if blockedApps[existingIndex].status == .dismissed {
            // 按最高置信来源判断
            let hasRealtimeSignal = event.sources.contains(.logStream)
                || event.sources.contains(.fsEvents)
            if !hasRealtimeSignal {
                // 仅来自 scan → 不重置 dismissed
                GLLog.state.debug("Scan result for dismissed app, skip: \(event.appName)")
                return
            }
            // 来自实时检测 → 重置为 blocked
            blockedApps[existingIndex].status = .blocked
            // ...
        }
    }
}
```

### 3.7 日志增强

| 位置 | 日志内容 | 级别 |
|------|---------|------|
| GatekeeperAssessor | `GK assess: <appName>, exitCode=<N>, result=accepted/rejected/unknown` | info |
| FSEventsWatcher 实时 | `FSEvents detected + assess: <appName>, result=<X>` | info |
| Fallback scan 候选 | `Fallback scan: <N> recent candidates` | info |
| Fallback scan 冷却跳过 | `Fallback scan skipped: cooldown (<N>s remaining)` | debug |
| LogStreamMonitor GK 行 | 原始行内容（开发阶段用 `.public` privacy） | debug |
| addBlockedApp scan 跳过 | `Scan result for dismissed app, skip: <appName>` | debug |

### 3.8 与分级权限模型兼容

| 权限级别 | scan 范围 | 触发条件 |
|---------|----------|---------|
| Level 0（无 TCC） | 仅 `/Applications`（非 TCC 保护） | Prompt shown + 近期候选 |
| Level 1（用户授权后） | + `~/Downloads`、`~/Desktop` | 同上，范围扩大 |

> 本次修复不改变权限模型本身，仅确保 scan 范围在各 Level 下正确运行。

## 六、风险评估

| 风险 | 严重性 | 缓解措施 |
|------|--------|---------|
| `spctl --assess` 对某些 app 执行慢 | 低 | 异步执行不阻塞 UI；目标化 scan 限制调用次数 |
| `spctl --assess` 触发 syspolicyd 日志反馈循环 | **高** | `evaluateScanResult` 不触发 scan、120s 冷却 |
| `spctl --assess` 在 macOS 未来版本行为变化 | 中 | Channel A 不受影响；`--assess` 是仅存未废弃的 spctl 子命令 |
| `spctl --assess` 对未签名 app 可能弹系统对话框 | 中 | 需用户先在终端验证（见下方待确认） |
| 收窄 `onGKActivity` 后漏掉某些拦截 | 低 | Channel A 是主检测通道，不受此修改影响 |

## 七、实测验证结果（2026-03-03 已确认 ✅）

> 以下为真机实测结果，**方案已确认可行**。

**验证 1**：正常 quarantined app（Sublime Text）

```bash
$ spctl --assess --type exec "/Applications/Sublime Text.app" 2>&1; echo "exit: $?"
exit: 0   # ← accepted，无弹窗 ✅
```

**结论**：Sublime Text 有 quarantine 属性但 GK 判定 accepted → 当前 GreenLight 标记为 BLOCKED 确实是**误报**。

**验证 2**：签名异常的开发构建（Antigravity Tools）

```bash
$ spctl --assess --type exec "/Applications/Antigravity Tools.app" 2>&1; echo "exit: $?"
/Applications/Antigravity Tools.app: code has no resources but signature indicates they must be present
exit: 1   # ← general failure（非 rejected），无弹窗 ✅
```

**结论**：exit 1 = 签名完整性问题（开发构建常见），三态模型归为 `unknown` → 仅记录日志不入 blocked，由 Channel A 在用户实际打开时精确捕获。

**综合确认**：
- `spctl --assess` 不会触发系统弹窗 ✅
- 退出码语义与 `man spctl` 文档一致 ✅
- 三态模型（0=accepted / 3=rejected / 其他=unknown）必要且充分 ✅

## 八、Sentinel 参考说明

Sentinel（alienator88）使用 `spctl --status / --global-enable / --global-disable` 管理 GK **全局开关**，这与本方案使用 `spctl --assess` 进行 **per-app 判定**是不同层面的用法。本方案借鉴了 spctl 作为 GK 状态查询工具的**思路**，但实现路径不同。
