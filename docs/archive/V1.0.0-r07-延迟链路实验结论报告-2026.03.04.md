## 二、核心发现：syspolicyd 原始日志结构

### 2.1 首次可见的 syspolicyd 日志格式

修复 `<private>` 后，首次看到 syspolicyd 原始日志的完整结构：

**performScan（扫描开始）**：
```
GK performScan: PST: (path: <hash>), (team: <team_id|null>), (id: <null>), (bundle_id: <null>)
```

**evaluateScanResult（扫描结果）**：
```
GK evaluateScanResult: <result_code>, PST: (path: <hash>), (team: <team_id|null>), (id: <code_id>), (bundle_id: <bundle_id>), <7个数字参数>
```

**XprotectFramework 连接**：
```
activating connection: mach=false listener=false peer=false name=com.apple.XprotectFramework.AnalysisService
```

**Stale cache（缓存过期）**：
```
Discovered stale cache entry for quarantined item: PST: (path: <hash>), ...
```

### 2.2 关键发现：path 是哈希值，不是真实路径

> [!CAUTION]
> **syspolicyd 中的 `path` 字段是哈希值（如 `3e00aa41bc9baaf`），不是文件系统路径！**

这意味着：
- ❌ 我们**无法**从 `performScan` 的 path 直接定位 APP
- ✅ 但同一 APP 的多次扫描会产生**相同哈希**，可用于关联
- ✅ `evaluateScanResult` 中的 **`id`** 和 **`bundle_id`** 是**明文**

### 2.3 evaluateScanResult 的 result_code 含义

| result_code | 出现场景 | 含义推断 |
|-------------|---------|---------|
| **0** | TestUnsigned, TestAdHoc, TestDamaged, Antigravity Tools | **被拦截**（GK 认定无法通过） |
| **2** | TestNormal, Sublime Text, CC Switch 2 | **通过/已信任**（GK 认定可以运行） |

> 这是**极其重要的早期信号**：`evaluateScanResult` 的 **result_code = 0** 就意味着 GK 会拦截。

### 2.4 evaluateScanResult 末尾 7 个参数对比

```
TestUnsigned:       1, 0, 1, 0, 8, 0, 0  ← result=0 (blocked)
TestAdHoc:          1, 0, 1, 0, 0, 0, 0  ← result=0 (blocked)
TestDamaged:        1, 0, 1, 0, 11, 0, 0 ← result=0 (blocked)
TestNormal:         0, 0, 1, 0, 0, 0, 0  ← result=2 (passed)
Antigravity Tools:  1, 0, 1, 0, 8, 0, 0  ← result=0 (blocked)，与 TestUnsigned 完全一致
Sublime Text:       1, 1, 1, 0, 9, 0, 0  ← result=2 (passed)
CC Switch 2:        1, 1, 1, 0, 7, 7, 0  ← result=2 (passed)
```

推断各位含义：
- **第 1 位**：`1` = 有 quarantine，`0` = 无 quarantine（TestNormal 为 0，其余均 1）
- **第 2 位**：`1` = 签名验证通过，`0` = 签名失败/缺失（Sublime/CCSwitch 为 1，测试 APP 为 0）
- **第 3 位**：始终 `1`（未知，可能是固定标志）
- **第 4 位**：始终 `0`（未知）
- **第 5 位**：疑似 GK 策略检查位图（`8` = unsigned, `0` = adhoc, `11` = damaged, `9`/`7` = 正常）
- **第 6-7 位**：通常 `0`，CC Switch 2 有 `7, 0`（特殊？）

### 3.2 关于 flags 值的含义

| flags 值 | 二进制 | 出现 APP | 推断含义 |
|---------|--------|---------|---------|
| **131074** | `100000000000000010` | TestUnsigned, Antigravity Tools | **adhoc + 某标志位**（可能是 linkerSigned） |
| **2** | `10` | TestAdHoc, TestDamaged | **adhoc 签名**（标准 ad-hoc） |

> `131074 = 0x20002`：bit 1 = `kSecCodeSignatureAdhoc`(2)，bit 17 = `kSecCodeSignatureLinkerSigned`(131072)。
> TestUnsigned 编译时 swiftc 自动添加了 linker-signed ad-hoc 签名。Antigravity Tools 也是类似情况。

### 3.3 ⚠️ TestDamaged 的意外发现

> [!WARNING]
> **TestDamaged（篡改签名 APP）的 `SecStaticCodeCheckValidity` 居然通过了！**
>
> 预期：SecStaticCode VALIDITY 失败 → 直接判定 rejected
> 实际：SecStaticCode passed → 走到 spctl → exitCode=3 rejected
>
> 这意味着 `SecStaticCodeCheckValidity(code, [], nil)` 使用默认 flags **不检查资源完整性**。篡改的是**可执行文件末尾追加垃圾字节**，`SecStaticCodeCheckValidity` 默认只验证代码签名结构，不做 full-resource 校验。
>
> **影响**：某些类型的签名篡改无法被 SecStaticCode 快速排除，必须依赖 spctl 兜底。

## 四、核心问题回答

### 4.1 「当前能确认被 GK 拦截的 APP 是谁」？

**✅ 是的，能完整确认。** 有三层信息来源：

| 层级 | 信息来源 | 可用时机 | 可确认的信息 | 可靠性 |
|------|---------|---------|-----------|--------|
| **L1** | syspolicyd `evaluateScanResult` | GK 处理中（USER_MARK 后 ~1-4s） | `id` + `bundle_id` + `result_code` + `team` | ⭐⭐⭐⭐ 几乎所有被检查的 APP 都有 |
| **L2** | FSEvents + GatekeeperAssessor | GK 处理完后（USER_MARK 后 ~5-10s） | 完整路径 + app 名 + bundleId + 签名详情 | ⭐⭐⭐⭐⭐ 100% 可靠 |
| **L3** | Quarantine xattr | 同上 | 下载来源（浏览器 + UUID） | ⭐⭐⭐⭐ 辅助信息 |

**关键结论**：

1. **`evaluateScanResult` 中的 `id` 和 `bundle_id` 是最早可用的 APP 身份信号**——在 FSEvents 到达前 ~4-6 秒就已可见
2. **`result_code = 0` 表示被拦截**，`= 2` 表示通过——这比 FSEvents 更早、更直接
3. **但 `performScan` 中的 path 是哈希值**，无法用于文件系统定位——只有 `evaluateScanResult` 才包含有用信息

### 4.2 「谁触发了安全检测」？

**✅ 能确认。**

- **被拦截的 APP（result=0）**：syspolicyd 日志中直接包含 `id` 和 `bundle_id`
- **通过的 APP（result=2）**：同样包含 `id` 和 `bundle_id`（如 TestNormal: id=com.test.normal）
- **例外**：Sublime Text 有 `team=Z6D26JE4Y4` 但 id/bundle_id 均为 null；CC Switch 2 全部为 null

### 4.3 GK 事件模式差异

| APP 类别 | GK 事件模式 | 特征 |
|---------|-----------|------|
| **blocked app（首次扫描）** | `scan → other(XProtect) → evaluate(result=0)` | **3 事件三连**，间隔 ~500-900ms + ~80-105ms |
| **已信任 app（有 quarantine）** | `scan → other(XProtect) → evaluate(result=2)` | **同样 3 事件三连**（TestNormal） |
| **已信任 app（有缓存）** | `evaluate(result=2)` 单独 1 个 | **仅 1 个 eval**（Sublime Text, CC Switch 2） |
| **blocked app（stale cache）** | `other(stale) → scan → other(XProtect) → evaluate(result=0)` | **4 事件**（Antigravity Tools） |

> [!IMPORTANT]
> **重大发现**：blocked app 和 normal app **在 GK 事件模式上没有差异**（都是 scan→other→eval 三连）。
> 差异在于 `evaluateScanResult` 的 **result_code**：`0` = blocked，`2` = passed。
>
> 这意味着：**仅靠 GK 事件模式（事件类型/数量/间隔）无法区分 blocked 和 normal app。必须解析 `evaluateScanResult` 的内容才能判定。**

### 4.4 早期信号可行性评估

| 信号源 | 可用时机 | 能否识别 APP | 能否判定 blocked | 实用性 |
|--------|---------|------------|----------------|--------|
| `performScan` | 最早（GK 开始） | ❌ path 是哈希 | ❌ | 仅知道 GK 在处理某 APP |
| XProtect 连接 | 500-900ms 后 | ❌ 无 APP 信息 | ❌ | 无用 |
| `evaluateScanResult` | ~1-1.5s 后 | ✅ id + bundle_id | ✅ result_code=0 | ⭐⭐⭐⭐⭐ **最佳早期信号** |
| FSEvents | ~5-10s 后 | ✅ 完整路径 | ✅ 经过验签 | ⭐⭐⭐⭐ 但太慢 |

> **结论：`evaluateScanResult` 中的 `result_code=0` + `bundle_id` 是最理想的早期信号。它能在 FSEvents 到达前 4-8 秒就告诉我们「哪个 APP 被 GK 拦截了」。**

## 五、延迟数据汇总

| 阶段 | 测量值 | 说明 |
|------|--------|------|
| USER_MARK → GK performScan | **4-10s** | macOS 内核处理时间，不可控 |
| performScan → evaluateScanResult | **0.6-1.0s** | GK 扫描 + XProtect 检查 |
| evaluateScanResult → FSEvents | **3-9s** | macOS 内核写入隔离区 + FS 通知延迟 |
| FSEvents → SecStaticCode | **2-6ms** | 我们的代码，极快 |
| SecStaticCode → spctl | **0 或 200ms** | 仅 SecStaticCode 通过时才走 spctl |
| spctl → Pipeline received | **~1.1s** | Dedup 窗口（1s） |

**如果利用 `evaluateScanResult` 早期信号，总延迟可从 ~12s 缩短到 ~5-6s。**

## 七、问题与改进建议

### 7.1 TestDamaged 的 SecStaticCode 未能检出

**问题**：TestDamaged 是在二进制末尾追加垃圾字节造成的「篡改」，但 `SecStaticCodeCheckValidity(code, [], nil)` 默认 flags 不检查资源完整性。

**考虑点**：
- 如果用 `SecCSFlags(rawValue: kSecCSCheckAllArchitectures | kSecCSStrictValidate)` 可能能检出
- 但更严格的 flags 可能导致**误判增多**和**验证时间增加**
- 当前有 spctl 兜底，实际效果未受影响（只是慢了 ~200ms）
- **建议**：不改。当前双层校验（SecStaticCode 快速排除 + spctl 兜底）已足够

### 7.2 evaluateScanResult 早期利用的方案

**当前状态**：syspolicyd 的 `evaluateScanResult` 在 GK 完成后 ~1s 到达 LogStream，包含 `result_code` 和 `bundle_id`。但当前代码只分类记录事件类型，**没有利用 result_code 做早期判定**。

**可利用的优化路径**：
1. 在 `parseLogLine()` 中解析 `evaluateScanResult` 的 result_code
2. 当 result_code = 0 时，结合 `id` 和 `bundle_id`，提前触发检测流程
3. 省去等待 FSEvents 的 ~5-10s 延迟

**但要注意**：
- syspolicyd 日志中的 `path` 是**哈希值**，无法直接映射到文件系统路径
- `bundle_id` 可用于匹配已知 APP（L3 已检出的 blockedApps）
- 对**首次出现**的 APP，仍需等待 FSEvents 提供完整路径
- `evaluateScanResult` 中 id=null 或 bundle_id=null 的情况存在（Sublime Text, CC Switch 2）

### 7.3 快捷键被系统拦截的问题

**问题**：`Cmd+Shift+3/4/5/6` 是 macOS 截图快捷键，无法用于 USER_MARK。

**建议**：如果需要再次实验，改用 `Ctrl+Option+数字` 或 `Cmd+Option+数字` 组合键。

### 7.4 proactiveScan 耗时日志仍有 `<private>`

**观察**：`proactiveScan: 0 found, <private>ms` 中的耗时仍被遮蔽。原因是 `proactiveScan` 的日志在 `FSEventsWatcher.swift` 第 219 行的 `elapsed` 变量未加 `privacy: .public`。

## 八、最终结论

### ✅ 已验证

1. **APP 身份可完整确认**：通过 `evaluateScanResult` 的 `id`/`bundle_id`（早期）和 FSEvents 的完整路径（后期），能 100% 确认被 GK 拦截的是谁
2. **evaluateScanResult 是最有价值的早期信号**：result_code=0 = blocked，比 FSEvents 早 ~5-10 秒
3. **GK 事件模式不能区分 blocked/normal app**：都是 scan→other→eval 三连（首次扫描时），差异仅在 result_code
4. **Quarantine xattr 提供下载来源信息**：包括浏览器（Safari/Chrome）和下载时间
5. **签名详情（SecCodeCopySigningInformation）能区分签名类型**：通过 flags 值区分 adhoc(2) / linker-signed(131074)

### ⚠️ 局限与注意

1. **syspolicyd path 是哈希值**：无法反向定位文件，只能用 id/bundle_id 间接关联
2. **已缓存的 APP 只产生单个 eval 事件**：不走完整 scan 流程，信息更少（如 Sublime 的 id/bundle 均为 null）
3. **TestDamaged 的篡改方式暴露了 SecStaticCode 默认 flags 的盲区**：追加字节不会导致签名检查失败
4. **部分 APP 的 evaluateScanResult 缺少 id/bundle_id**（如 Sublime Text）：这些 APP 无法通过早期信号识别
