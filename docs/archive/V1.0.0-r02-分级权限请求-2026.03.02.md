## 一、问题背景

### 1.1 现状

当前 `GreenLightApp.init()` 无条件调用 `setupPipeline()`，启动时立即监控 3 个目录：

| 目录 | TCC 保护 | TCC 服务名 | 启动时是否触发权限弹窗 |
|------|----------|-----------|----------------------|
| `/Applications` | ❌ 不受保护 | — | 不会 |
| `~/Downloads` | ✅ Files & Folders | `kTCCServiceSystemPolicyDownloadsFolder` | **会** |
| `~/Desktop` | ✅ Files & Folders | `kTCCServiceSystemPolicyDesktopFolder` | **会** |

> [!IMPORTANT]
> macOS TCC 对 Desktop 和 Downloads 是**独立管理**的，访问两个目录会弹**两个单独的系统弹窗**，用户可以 Allow 一个、Deny 另一个。

### 1.2 问题

1. **体验割裂**：用户可能还在 Onboarding，系统突然弹出 TCC 权限弹窗，完全脱离上下文
2. **弹窗浪费**：TCC 弹窗只弹一次，用户若不理解用途而点了"Don't Allow"，后续只能去系统设置手动改
3. **不合时宜**：即使 Onboarding 已完成，后台 `onGKActivity` 兜底扫描也会在任意时刻触发弹窗（实际已复现）

### 1.3 目标

**场景驱动、分级拉起**——在用户有心理预期的上下文中请求权限，最大化授权通过率。

## 二、核心设计：两级管线

### 2.1 分级模型

```
┌───────────────────────────────────────────────────────────┐
│  Level 0 · 基础守护（零权限，App 启动即生效）              │
│                                                           │
│  ┌─────────────────────┐  ┌──────────────────────┐       │
│  │  LogStreamMonitor   │  │  FSEventsWatcher     │       │
│  │  (Channel A)        │  │  (Channel B)         │       │
│  │                     │  │                      │       │
│  │  监听 syspolicyd    │  │  仅监控 /Applications │       │
│  │  日志流             │  │  零 TCC 依赖          │       │
│  └──────────┬──────────┘  └──────────┬───────────┘       │
│             └──────────┬─────────────┘                    │
│                        ▼                                  │
│             EventDeduplicator → 通知 → 修复               │
└───────────────────────────────────────────────────────────┘

┌───────────────────────────────────────────────────────────┐
│  Level 1 · 增强守护（需 TCC 授权，场景驱动拉起）           │
│                                                           │
│  ┌──────────────────────────────────────────┐             │
│  │  FSEventsWatcher 按目录独立追加           │             │
│  │  + ~/Downloads  (TCC: 独立授权)          │             │
│  │  + ~/Desktop    (TCC: 独立授权)          │             │
│  │  允许部分授权：只要任一目录可用即追加      │             │
│  └──────────────────────────────────────────┘             │
└───────────────────────────────────────────────────────────┘
```

### 2.2 Level 0 的覆盖能力

> [!IMPORTANT]
> Level 0 **并非残缺模式**，它能覆盖绝大多数实际场景。

| 场景 | Level 0 能否覆盖 | 覆盖机制 |
|------|:---:|------|
| 用户双击被拦截 app（任意位置） | ✅ | Channel A 捕获 syspolicyd 日志 → 提取 `file://` 路径 |
| 从 DMG 拖入 /Applications 后运行 | ✅ | Channel A + Channel B (FSEvents /Applications) |
| App 更新后重新被隔离 | ✅ | Channel A 捕获 GK 检查 |
| 在 ~/Downloads 中有 app 但用户从未双击 | ❌ | 无法主动发现（无 GK 日志触发） |
| 在 ~/Desktop 上放了 app 但从未运行 | ❌ | 同上 |

**遗漏场景的概率极低**：只有"用户下载了 app 但从未试图打开"时 Level 0 才无法发现，这恰恰不是 GreenLight 的核心场景（GreenLight 解决的是"打开被拦截"的问题）。

### 2.3 Level 1 的增量价值

| 增量能力 | 说明 |
|----------|------|
| **主动扫描** | 手动扫描时能枚举 ~/Downloads 和 ~/Desktop 中所有带 quarantine 的 .app |
| **GK 兜底补全** | `onGKActivity` 触发时可扫描 TCC 保护目录，减少 Channel A 路径提取失败的遗漏 |
| **FSEvents 实时监控** | 用户在 ~/Downloads、~/Desktop 中增删 app 时实时检测 |

## 三、场景驱动的授权交互

### 3.1 触发时机

Level 1 授权**仅在以下两个场景中**被触发：

| # | 触发场景 | 触发条件 | 冷却约束 | 说明 |
|---|----------|----------|----------|------|
| **T1** | **GK 活动兜底**（被动） | 置信度门槛达标（见 §3.6） | ✅ 受冷却限制 | LogStream 检测到 GK 活动但无法提取完整路径 |
| **T2** | **用户手动操作**（主动） | 用户点击扫描按钮或 "Expand Coverage" | ❌ 不受冷却限制 | 用户明确意图，永远立即响应 |

> [!TIP]
> **不主动骚扰用户**。如果 Level 0 一直工作良好（Channel A 总能提取到完整路径），用户可能永远不会看到 Level 1 的授权引导——这是正确的行为。

### 3.2 授权引导面板（Menu Bar 浮动面板）

当触发条件满足时，GreenLight 在 **Menu Bar 下方弹出浮动面板**（复用 `DetectionPanelController` 的定位逻辑），弹出位置与检测通知面板一致，保持用户认知一致性。

```
┌──────────────────────────────────────────────────┐
│                                                  │
│   🔍  Enhance Protection?                        │
│                                                  │
│   GreenLight detected Gatekeeper activity but    │
│   couldn't fully verify the file location.       │
│                                                  │
│   Grant access to Downloads and Desktop for      │
│   comprehensive monitoring — files are only      │
│   scanned locally, nothing is uploaded or shared. │
│                                                  │
│   ┌────────────────────────────────────────┐     │
│   │  📁  ~/Downloads                       │     │
│   │  📁  ~/Desktop                         │     │
│   └────────────────────────────────────────┘     │
│                                                  │
│        [ Not Now ]         [ Grant Access ]       │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 3.3 交互流程

```
T1(置信度达标) 或 T2(用户主动) 触发
        │
        ▼
  任一 TCC 目录已有权限？ ──全部已有──→ 升级到 Level 1（或已在 Level 1）
        │
    部分/全无
        │
        ▼
  T1 触发？
   ├── Yes → 冷却中？ ──Yes──→ 静默降级
   │            │
   │           No
   │            │
   └── No (T2) ─┘
        │
        ▼
  显示 Menu Bar 浮动引导面板
        │
        ├── [ Not Now ] ──→ 标记本次会话已拒绝（仅约束 T1）
        │                    继续 Level 0 运行
        │
        └── [ Grant Access ] ──→ 依次触发未授权目录的文件访问
                                  ├── ~/Downloads → 系统 TCC 弹窗
                                  │   ├── Allow → 追加到监控
                                  │   └── Deny  → 跳过，继续下一个
                                  │
                                  └── ~/Desktop → 系统 TCC 弹窗（如未授权）
                                      ├── Allow → 追加到监控
                                      └── Deny  → 跳过
                                  │
                                  ▼
                            至少一个 Allow？
                              ├── Yes → 补充扫描已授权目录
                              └── No  → 全部被拒，不再弹窗
```

### 3.4 文案规格

| 元素 | 英文 | 设计理由 |
|------|------|----------|
| **标题** | Enhance Protection? | 问句，非要求，降低压迫感 |
| **原因** | GreenLight detected Gatekeeper activity but couldn't fully verify the file location. | **先讲事实（有活动），再讲局限（无法完全验证）** |
| **请求** | Grant access to Downloads and Desktop for comprehensive monitoring — files are only scanned locally, nothing is uploaded or shared. | 明确范围 + 可维护的隐私承诺（避免"ever"绝对措辞） |
| **拒绝 CTA** | Not Now | 暗示"以后可能会需要"，保留余地 |
| **确认 CTA** | Grant Access | 明确动作 |

### 3.5 弹窗防骚扰规则

| 规则 | 适用范围 | 说明 |
|------|----------|------|
| **会话级节流** | 仅 T1（被动） | 用户点击 "Not Now" 后，本次 App 运行期间 T1 不再触发引导 |
| **全局冷却** | 仅 T1（被动） | 即使重启 App，24 小时内 T1 只触发一次 |
| **主动操作豁免** | T2（主动） | 用户点击扫描/Expand Coverage 永远不受冷却限制，立即进入授权流程 |
| **全部拒绝后静默** | T1 + T2 | 如果所有目录的 TCC 都被 Deny（实时探测确认），不再弹引导面板，转为 Popover 中的设置提示 |

### 3.6 T1 置信度门槛

为避免噪声日志（如 `<private>` 标记的无关 GK 内部操作）触发不必要的引导弹窗，T1 设置置信度门槛：

| 参数 | 值 | 说明 |
|------|-----|------|
| **窗口时长** | 10 秒 | 从首个 `onGKActivity` 开始计时 |
| **最少 GK 活动次数** | ≥ 2 次 | 同一窗口内至少 2 次无路径的 GK 活动 |
| **成功检测为零** | 0 次 `DetectionEvent` | 窗口内 Channel A 未能提取到任何有效路径 |

**仅当三个条件同时满足时**，才判定为"可能有遗漏"并触发 Level 1 引导。

## 四、权限状态管理

### 4.1 核心原则：系统实时探测为唯一信源

> [!IMPORTANT]
> **App 不维护权限状态副本**。每次需要判断目录可用性时，实时探测系统真实状态。
> App 只持久化**用户的意愿偏好**（"我不想被问"），而非系统的权限状态。

这样设计的好处：
- 用户在系统设置中事后改权限 → App 下次探测时自动感知，无需同步
- 不存在"App 记录已授权但实际已撤销"的不可恢复僵局

### 4.2 TCC 授权探测方法

macOS 没有直接查询 "Files & Folders" TCC 权限的 API。采用**试探法**，按目录独立探测：

```swift
/// 检查是否有目录的读取权限（通过实际尝试枚举）
/// - 有权限：返回内容数组（即使为空）
/// - 无权限：抛出 NSCocoaErrorDomain Code 257 "Permission denied"
static func canAccessDirectory(_ url: URL) -> Bool {
    let fm = FileManager.default
    do {
        _ = try fm.contentsOfDirectory(at: url, includingPropertiesForKeys: nil)
        return true
    } catch {
        return false
    }
}
```

### 4.3 部分授权支持

Desktop 和 Downloads 是独立 TCC 服务，用户可能 Allow 一个、Deny 另一个。**GreenLight 支持部分授权**：

```swift
func upgradeToLevel1() {
    var grantedDirs: [URL] = []
    
    for dir in FSEventsWatcher.level1Directories {
        if canAccessDirectory(dir) {
            grantedDirs.append(dir)
        }
    }
    
    if !grantedDirs.isEmpty {
        fsWatcher.addDirectories(grantedDirs)
        let events = fsWatcher.scanApps(in: grantedDirs)
        for event in events { deduplicator.receive(event) }
    }
}
```

### 4.4 持久化字段

App 只持久化**用户意愿**，不持久化权限状态：

| Key | 类型 | 默认值 | 说明 |
|-----|------|--------|------|
| `greenlight.lastEnhancePromptDate` | `Date?` | `nil` | 上次 T1 弹出引导面板的时间（24h 冷却） |

### 4.5 会话级状态（不持久化）

| 属性 | 类型 | 说明 |
|------|------|------|
| `sessionDismissedEnhancePrompt` | `Bool` | 本次运行是否已点击 "Not Now"（仅约束 T1） |

## 五、管线启动逻辑重构

### 5.1 新逻辑

```swift
private func setupPipeline() {
    // === Level 0: 零权限通道，无条件启动 ===
    
    // Channel A: LogStream（不需要任何文件权限）
    logMonitor.startMonitoring()
    
    // Channel B (Level 0): 仅监控 /Applications（无 TCC）
    fsWatcher.startWatching(directories: FSEventsWatcher.level0Directories)
    
    // === Level 1: 启动时实时探测已授权目录 ===
    
    let grantedLevel1Dirs = FSEventsWatcher.level1Directories.filter {
        FSEventsWatcher.canAccessDirectory($0)
    }
    if !grantedLevel1Dirs.isEmpty {
        fsWatcher.addDirectories(grantedLevel1Dirs)
    }
    
    // ... 事件路由设置（不变）...
    
    // onGKActivity 回调：置信度计数 + Level 1 引导
    logMonitor.onGKActivity = { [weak self] in
        self?.recordGKActivity()
    }
}
```

### 5.2 T1 置信度窗口实现

```swift
private var gkActivityWindow: (start: Date, count: Int, hasDetection: Bool)?

private func recordGKActivity() {
    let now = Date()
    
    if let window = gkActivityWindow,
       now.timeIntervalSince(window.start) < 10 {
        // 在窗口内，累加
        gkActivityWindow?.count += 1
    } else {
        // 新窗口
        gkActivityWindow = (start: now, count: 1, hasDetection: false)
        
        // 窗口到期检查
        DispatchQueue.main.asyncAfter(deadline: .now() + 10) { [weak self] in
            self?.evaluateGKWindow()
        }
    }
    
    // 无论如何，先执行 Level 0 兜底扫描
    performFallbackScan(includeLevel1: false)
}

private func evaluateGKWindow() {
    guard let window = gkActivityWindow else { return }
    
    // 置信度门槛：≥2 次 GK 活动 + 0 次成功检测
    if window.count >= 2 && !window.hasDetection {
        showEnhanceProtectionPanel()
    }
    
    gkActivityWindow = nil
}
```

## 六、FSEventsWatcher 改造

### 6.1 目录分级

```swift
static let level0Directories: [URL] = [
    URL(fileURLWithPath: "/Applications")
]

static let level1Directories: [URL] = [
    URL.homeDirectory.appending(path: "Downloads"),
    URL.homeDirectory.appending(path: "Desktop")
]
```

### 6.2 新增方法

```swift
extension FSEventsWatcher {
    /// 动态追加监控目录（Level 1 升级时调用）
    func addDirectories(_ newDirs: [URL]) {
        stopWatching()
        let currentDirs = currentMonitoredDirectories
        let allDirs = currentDirs + newDirs.filter { !currentDirs.contains($0) }
        startWatching(directories: allDirs)
        GLLog.fsEvents.notice("Directories upgraded: \(allDirs.map(\.path))")
    }
    
    /// TCC 权限探测
    static func canAccessDirectory(_ url: URL) -> Bool {
        do {
            _ = try FileManager.default.contentsOfDirectory(
                at: url, includingPropertiesForKeys: nil
            )
            return true
        } catch {
            return false
        }
    }
}
```

## 七、Info.plist 配置

> [!IMPORTANT]
> 必须配置 usage description，否则系统 TCC 弹窗上缺少解释文案，降低授权通过率。

| Key | 值 |
|-----|-----|
| `NSDesktopFolderUsageDescription` | GreenLight monitors your Desktop for apps blocked by macOS Gatekeeper, so it can detect and help you fix them instantly. |
| `NSDownloadsFolderUsageDescription` | GreenLight monitors your Downloads for apps blocked by macOS Gatekeeper, so it can detect and help you fix them instantly. |

文案设计要点：与 App 内引导面板文案形成一致链路——用户先看到 App 内解释（"Enhance Protection?"），再看到系统弹窗解释（同样提到 Gatekeeper + 检测 + 修复），认知连贯。

## 十一、测试策略

### 11.1 可自动化测试

| 测试项 | 覆盖点 |
|--------|--------|
| 目录分级计算 | `level0Directories` / `level1Directories` 正确性 |
| T1 置信度窗口 | 窗口内计数、窗口到期时达标/未达标 |
| 冷却逻辑 | T1 被动触发受冷却约束、T2 主动触发不受冷却 |
| `lastEnhancePromptDate` 读写 | 24h 冷却计时正确性 |
| 部分授权逻辑 | 给定不同的 `canAccessDirectory` 返回值，验证追加正确目录 |

### 11.2 需手动测试

| 实际 TCC 弹窗流程 | 无法在 CI 中模拟系统 TCC 弹窗 |

| 系统设置中改权限后的回读 | 需手动操作系统设置面板 |
| 引导面板定位和视觉 | UI 验收 |
