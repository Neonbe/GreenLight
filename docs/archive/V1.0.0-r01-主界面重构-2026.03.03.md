## 一、重构动机

### 1.1 现状问题

当前 `PopoverView.swift` 是为 **Menu Bar Popover (420×380)** 设计的紧凑面板。转为 Dock App 主窗口后产生以下问题：

| # | 问题 | 严重度 |
|---|------|--------|
| P1 | **窗口与 Onboarding 尺寸不一致** — Onboarding 900×620，主界面 420×380 | 高 |
| P2 | **状态语义不精确** — 黄灯仅在 `isScanning` 时显示，不符合业务实际（应表示"未决策"） | 高 |
| P3 | **空间浪费** — 黄灯区仅扫描时出现，大部分时间只有红+绿两个 Lane | 高 |
| P4 | **信息密度与窗口比例失衡** — 少量内容挤在小区域，其余全是空白 | 高 |
| P5 | **System Protected 无实际功能** — 底部状态胶囊占位但无用 | 中 |
| P6 | **字体风格不统一** — 主界面用 `.bold/.heavy`，与 Onboarding `.light` 风格断裂 | 中 |
| P7 | **设置按钮位置过于突出** — 与扫描按钮并排在 Header 右侧，抢占视觉焦点 | 中 |
| P8 | **列表超长内容无处理方案** — App 数量多或名称长时无截断/折叠策略 | 中 |
| P9 | **设置页过于简陋** — 仅有一个 Toggle + 两行文本，不像正式产品 | 中 |
| P10 | **缺少用户反馈通道** — 没有"联系我们"入口 | 低 |
| P11 | **动画体系未复用** — 主界面未使用 `NDAnimation` 弹簧物理引擎 | 低 |
| P12 | **Hover/交互反馈不足** — Lane 和图标缺少完善的 hover 态视觉反馈 | 低 |

### 1.2 设计目标

| 目标 | 验收标准 |
|------|----------|
| **状态语义精确** | 三灯定义与业务逻辑一一对应，无歧义 |
| **空间利用** | 三区始终可见，无论数据多少都不浪费空间 |
| **视觉一致** | 主界面与 Onboarding 风格统一：900×620、hiddenTitleBar、`.light` 字体 |
| **信息层次** | 一眼看到全局状态，按需下钻到具体 App |
| **品牌强化** | 红绿灯作为视觉锚点始终在位 |
| **可扩展** | 设置页、联系入口等功能可平滑扩展 |

## 二、信号灯语义重新定义

> [!IMPORTANT]
> 这是核心业务变更，三灯语义与旧版完全不同。

### 2.1 新版语义

| 信号 | 颜色值 | 标签 | 定义 | 数据来源 |
|:---|:---|:---|:---|:---|
| 🟢 **绿灯** | `#22C55E` | `CLEARED` | 用户通过 GreenLight **手动 Grade**（放行）的应用 | `AppRecord.status == .cleared` |
| 🟡 **黄灯** | `#F59E0B` | `UNRESOLVED` | 出现了但**没有被用户决策**的应用（pending/未处理） | `AppRecord.status == .pending` 或 `dismissed` |
| 🔴 **红灯** | `#EF4444` | `REJECTED` | 在扫描/检测过程中**被系统拒绝（Gatekeeper 拦截）** 的应用 | `AppRecord.status == .blocked` |

### 2.2 与旧版对比

| 维度 | 旧版 | 新版 |
|------|------|------|
| 🟢 绿灯 | "已放行" (cleared) | **不变**，更明确为"用户手动 Grade" |
| 🟡 黄灯 | "扫描中" (isScanning) | 改为 **"未决策"**（有事件但用户未处理） |
| 🔴 红灯 | "被拦截" (blocked) | 改为 **"被拒绝"**（被 Gatekeeper 拦截的应用） |

### 2.3 状态流转

```
                应用首次出现
                    │
                    ▼
              🟡 UNRESOLVED
          ┌─────┴──────────────┐
          │                    │
    用户"放行"            用户"忽略"(dismiss)
          │                    │
          ▼                    ▼
    🟢 CLEARED          留在 🟡（不消失,
          │              提示"已暂时忽略"）
          │
    更新后被重新拦截
          │
          ▼
    🔴 REJECTED
          │
    用户再次放行
          │
          ▼
    🟢 CLEARED
```

> [!NOTE]
> `dismissed` 状态在旧版归入红灯区（与 blocked 混合）。新版中 **dismissed 归入黄灯区**，表示"用户看到了但没有做出最终决策"。

### 2.4 对 AppRecord.Status 的影响

现有枚举 `pending → blocked → dismissed → cleared` 不需要修改。改变的是 **UI 映射**：

| Status | 旧版 Lane | 新版 Lane |
|--------|----------|----------|
| `.pending` | 黄灯（仅扫描时） | 🟡 黄灯（始终） |
| `.blocked` | 🔴 红灯 | 🔴 红灯 |
| `.dismissed` | 🔴 红灯（混入） | 🟡 黄灯 |
| `.cleared` | 🟢 绿灯 | 🟢 绿灯 |

## 三、首页布局方案

### 3.1 整体架构

```
┌──────────────────────────────────────────────────────────────────┐
│ ●●●                                                              │
│                                                                   │
│    G R E E N L I G H T                              ↻            │
│                                                                   │
│    ┌────────────────────────────────────────────────────────┐     │
│    │                                                        │     │
│    │              ┌────────┐                                │     │
│    │              │        │                                │     │
│    │              │  🔴 ─────── REJECTED    {N}  [icons]   │     │
│    │              │        │                                │     │
│    │              │  🟡 ─────── UNRESOLVED  {N}  [icons]   │     │
│    │              │        │                                │     │
│    │              │  🟢 ─────── CLEARED     {N}  [icons]   │     │
│    │              │        │                                │     │
│    │              └──┬─────┘                                │     │
│    │                 │                                      │     │
│    └────────────────────────────────────────────────────────┘     │
│                                                                   │
│  ⚙ Settings                              累计亮绿灯 42 次        │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

### 3.2 关键设计决策

| 决策 | 说明 |
|------|------|
| **三区始终可见** | 红、黄、绿 Lane 永远显示，无数据时展示空状态文案，避免布局跳动 |
| **红绿灯居左，Lane 水平延伸** | 沿用 `ui_design_spec.md §3.1` 的锚点布局，放大至主窗口比例 |
| **Lane 等高分布** | 三个 Lane 等分内容区高度，保证视觉均衡 |
| **统一字体风格** | 全部采用 `.light/.regular` 字重，与 Onboarding 一致 |
| **居中布局** | 内容块整体居中，左右留白各 ~80px，上下留白充足 |
| **删除 System Protected** | 底部状态胶囊无实际功能，移除以减少视觉噪音 |
| **设置入口移至左下角** | 降低设置按钮的视觉权重，放在 footer 左下角，形如 `⚙ Settings` |
| **统计信息移至右下角** | "累计亮绿灯 N 次" 保留在 footer 右下角，作为品牌强化 |
| **新增联系入口** | 设置页内增加"联系我们"区域 |

### 3.3 删除与新增

| 删除 | 理由 |
|------|------|
| 底部 `System Protected` 状态胶囊 | 无实际功能，纯装饰占位 |
| Header 右侧 `⚙` 齿轮按钮 | 移至左下角，降低视觉权重 |

| 新增 | 说明 |
|------|------|
| 左下角 `⚙ Settings` 入口 | 文字 + 图标，视觉低调但可发现 |
| 设置页内"联系我们"区域 | 邮箱 + 链接，方便用户反馈 |

### 3.4 Lane 空状态文案

| Lane | 有数据 | 空状态 |
|------|--------|--------|
| 🔴 Red | `REJECTED  {N}` + App 图标 | `No rejected apps` |
| 🟡 Amber | `UNRESOLVED  {N}` + App 图标 | `All clear — nothing pending` |
| 🟢 Green | `CLEARED  {N}` + App 图标 | `No apps cleared yet` |

### 3.5 Expand Coverage 提示

- 位置：主内容区下方、Footer 上方
- 样式：蓝色底卡片，与当前一致
- 触发条件：扫描后发现未授权目录
- 出现时 Footer 整体下移，窗口高度不变（内部弹性吸收）

## 四、列表超长内容方案（待讨论）

> [!IMPORTANT]
> 以下三个方案各有利弊，需要一起讨论后选择一个。

当某个 Lane 中的 App 数量超出一行横向空间时，需要处理溢出。

### 方案 A：横向滚动 + 渐隐遮罩

```
┌─ Lane ─────────────────────────────────────────┐
│  REJECTED  3  │  🟦 🟧 🟪 ░░░▒▒▓              │  ← 右侧渐隐
│                                     ← swipe →  │
└─────────────────────────────────────────────────┘
```

- **当前实现**：已在 PopoverView 中使用 `ScrollView(.horizontal)`
- **优点**：布局稳定，Lane 高度不变，视觉一致
- **缺点**：隐藏了部分图标，用户可能不知道可以滚动
- **增强**：右侧添加 `LinearGradient` 渐隐遮罩，暗示可滚动

### 方案 B：折叠展开 + 计数角标

```
折叠态：
┌─ Lane ─────────────────────────────────────────┐
│  REJECTED  8  │  🟦 🟧 🟪 🟩 🟥  +3 ▾         │
└─────────────────────────────────────────────────┘

展开态：（仅激活的 Lane 展开，其他 Lane 压缩）
┌─ Lane ─────────────────────────────────────────┐
│  REJECTED  8  │  🟦 🟧 🟪 🟩 🟥                │
│               │  🟧 🟦 🟪                  ▴   │
└─────────────────────────────────────────────────┘
```

- **优点**：一眼看到总数，不丢失任何信息
- **缺点**：展开时 Lane 高度变化，可能造成布局跳动
- **增强**：使用 `NDAnimation.panelTransition` 弹簧动画过渡高度变化

### 方案 C：悬停弹出浮层

```
默认态：
┌─ Lane ─────────────────────────────────────────┐
│  REJECTED  8  │  🟦 🟧 🟪 🟩 🟥  +3            │
└─────────────────────────────────────────────────┘

Hover +3 后弹出：
┌──────────────┐
│  🟧 🟦 🟪    │  ← 浮层显示剩余图标
└──────────────┘
```

- **优点**：主布局完全不变，极致简洁
- **缺点**：需要 hover 操作才能看到全部，移动端无 hover
- **增强**：浮层使用 `popover` 或 `overlay` 实现

### 方案对比

| 维度 | A 横向滚动 | B 折叠展开 | C 悬停浮层 |
|------|-----------|-----------|-----------|
| 布局稳定性 | ★★★ 高 | ★★ 中 | ★★★ 高 |
| 信息可见度 | ★★ 中 | ★★★ 高 | ★ 低 |
| 实现复杂度 | ★ 低 | ★★ 中 | ★★ 中 |
| macOS 原生感 | ★★★ 高 | ★★ 中 | ★★★ 高 |
| 适合 GreenLight | 推荐 | 备选 | 不推荐 |

> 我的初步倾向是 **方案 A（增强版）**，理由：macOS 原生应用普遍使用横向滚动处理溢出（如 Launchpad、Finder 标签页），用户认知成本最低。右侧渐隐遮罩能有效暗示可滚动。

- 两行 LabeledContent（版本、累计绿灯数）
- 尺寸 450×300，作为 `.sheet` 弹出

问题：
1. 过于简陋，不像正式产品
2. 缺少反馈通道
3. 使用系统默认 Form 样式，与深色 Onboarding 风格不一致

## 七、动画体系

### 7.1 统一使用 NDAnimation

所有主界面动画必须复用 `NDAnimation` 弹簧预设：

| 场景 | 预设 | 参数 |
|------|------|------|
| 主界面入场 | `panelTransition` | response 0.35, damping 0.85 |
| App 图标入场 | `cardEnter` + 交错 | response 0.25, damping 0.6, 每个延迟 50ms |
| 页面切换（主界面 ↔ 设置） | `panelTransition` | 左右滑动 + opacity |
| Lane hover 响应 | `cardEnter` | response 0.25, damping 0.6 |
| 图标 hover (scale/offset) | Onboarding stagger 动画 | spring response 0.45, damping 0.78 |

### 7.2 入场编排

```
T=0ms    窗口淡入 + scale(0.98→1)        ← panelTransition
T=80ms   品牌名淡入                       ← cardEnter
T=130ms  红绿灯整体淡入                   ← cardEnter
T=180ms  Lane 1 (红) 从左侧滑入           ← cardEnter + stagger(0)
T=230ms  Lane 2 (黄) 滑入                 ← cardEnter + stagger(1)
T=280ms  Lane 3 (绿) 滑入                 ← cardEnter + stagger(2)
T=350ms  App 图标交错入场（每个 50ms）     ← cardEnter + stagger
T=450ms  Footer 淡入                      ← panelTransition
```

### 7.3 App 图标交互

| 状态 | 效果 | 动画 |
|------|------|------|
| Default | 原始尺寸 | — |
| Hover | `scale(1.15) translateY(-2px)` + 阴影加深 + 边框提亮 | spring 0.25/0.85 |
| Active (按下) | `scale(0.94)` | 150ms |
| 弹出 Popover | ActionBubble 出现 | panelTransition |

### 7.4 prefers-reduced-motion

全部动画必须检查 `@Environment(\.accessibilityReduceMotion)`：
- **开启时**：所有 spring 动画用 `nil` 或 0 时长替换
- **交错入场**：省略延迟，瞬间出现
- **hover 效果**：仅改变 opacity，不做 scale/offset

## 十、AI 自查 — 深度审视补充发现

以下是在调研过程中额外发现的问题和建议：

### 10.1 设计层面

| # | 发现 | 建议 |
|---|------|------|
| D1 | **色彩 Token 分散** — OB 枚举、DetectionPanel 私有常量、PopoverView 内联硬编码三套不同定义，颜色值其实一样 | 应抽取为全局 `DesignTokens` 共享 |
| D2 | **按钮样式分散** — `OBButtonStyle`、`ActionButtonStyle`、DetectionPanel 的 `panelButton` 三套独立实现 | 应统一为一个参数化的 `GLButtonStyle` |
| D3 | **红绿灯组件重复** — `PopoverView.trafficLight`（内联实现）和 `OBTrafficLight`（Onboarding 私有）是两套不同实现，渲染参数不同 | 应抽取为共享组件 `TrafficLightView`，通过参数控制尺寸 |
| D4 | **Lane 缺少 hover 反馈** — 当前 Lane 是静态卡片，无 hover 态 | 应增加 hover 态背景微亮 + 霓虹线加亮 |

### 10.2 交互层面

| # | 发现 | 建议 |
|---|------|------|
| I1 | **空列表缺少引导** — 新用户 Onboarding 完成后进入主界面，三个 Lane 全空，缺少上下文 | 空状态应有简短引导文案（如 "Apps detected by Gatekeeper will appear here"） |
| I2 | **扫描按钮无状态反馈** — 点击扫描按钮后仅 `isScanning = true`，按钮无旋转动画 | 应在扫描中给 `↻` 图标添加 `rotationEffect` 无限旋转 |
| I3 | **ActionBubble 使用 popover** — macOS native popover 有指向箭头但位置固定 | 当前实现可接受，但考虑升级为自定义 overlay 以获得更好的动画控制 |

### 10.3 性能层面

| # | 发现 | 建议 |
|---|------|------|
| F1 | **图标数据作为 Data 存储** — `appIcon: Data?` 每次渲染都要 `NSImage(data:)` 解码 | 可考虑图标缓存层，但 V1 数据量小可暂不处理 |
