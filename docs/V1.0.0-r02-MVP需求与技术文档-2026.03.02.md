## 一、产品定义

### 一句话定位

**GreenLight — macOS 后台智能守卫，自动检测被 Gatekeeper 拦截的应用并一键放行。**

### 核心隐喻

🚦 **红绿灯** — 为你的应用"放行"。

| 信号 | 含义 | 触发条件 |
|------|------|----------|
| 🟢 绿灯 | 已放行 | `xattr -rd` 成功执行 |
| 🔴 红灯 | 被拦截 | 检测到 quarantine 属性的 .app |
| 🟡 黄灯 | 等待中 | 扫描中 / 待用户决策 |

### 与 Sentinel 的定位差异

```
Sentinel = 手动挡跑车（精准但需要你自己踩油门）
GreenLight = 自动驾驶辅助（主动检测 + 辅助修复 + 仪表盘）
```

**我们只做 Sentinel 没做的事**。Sentinel 是一个优秀的手动工具（1,400+ Stars、40 万+下载），我们不与它在拖拽/右键上竞争，而是聚焦于它完全空白的领域：

| 能力 | Sentinel | GreenLight V1 |
|------|----------|---------------|
| 自动检测被拦截应用 | ❌ | ✅ **核心差异** |
| 主动通知推送 | ❌ | ✅ **核心差异** |
| 从通知一键修复 | ❌ | ✅ **核心差异** |
| Menu Bar 常驻守卫 | ❌ | ✅ |
| 首次快速扫描 | ❌ | ✅ |
| 拖拽 / 右键修复 | ✅ | ❌ 不做 |
| 自签名 | ✅ | ❌ 不做（V2） |

### 4.2 技术选型

| 层 | 技术 | 选型理由 |
|----|------|----------|
| **UI** | SwiftUI `MenuBarExtra` | macOS 原生 Menu Bar App 最佳实践 |
| **检测 A** | `Process` + `log stream` | 实验已验证可提取完整路径，精准触发 |
| **检测 B** | `FSEventStream` (CoreServices) | 实验已验证即时检测，稳定兜底 |

| **xattr 检查** | `getxattr` / `listxattr` C API | 直接调用 POSIX API 检查 quarantine 属性，零 shell 依赖 |
| **通知** | `UserNotifications` | macOS 原生通知，支持 Action Buttons |
| **自动打开** | `NSWorkspace.shared.open()` | Sentinel 已验证此模式 |

### 4.4 权限需求

| 权限 | 必需性 | 用途 | 用户感知 |
|------|--------|------|----------|
| **通知** | ✅ 必需 | 推送检测通知 | Onboarding 时一次性授权 |
| **Files & Folders (TCC)** | ✅ 必需 | 访问 `~/Downloads`、`~/Desktop` 进行 FSEvents 监控和 xattr 读写 | 系统自动弹窗，首次访问时授权。`/Applications` 不受 TCC 保护，无需额外授权 |
| **Full Disk Access** | 🟡 可选 | 读取 `/Applications` 以外更多目录的 xattr | 绝大部分场景不需要 |
| ~~Accessibility~~ | ❌ 不需要 | 已移除弹窗伴随设计 | — |
| ~~Automation~~ | ❌ 不需要 | 已移除 Terminal 注入 | — |
| ~~App Sandbox~~ | ❌ 不启用 | `log stream` 和 `xattr` 需要沙盒外运行 | GitHub/Homebrew 分发 |

**TCC 授权失败降级**：若用户拒绝 Downloads/Desktop 访问权限：
- FSEventsWatcher 对该目录的监控静默失效（无事件推送）
- `/Applications` 监控不受影响（不受 TCC 保护）
- Menu Bar Popover 展示提示条：「部分目录未获授权，点此前往系统设置」
- Channel A (LogStreamMonitor) 不受 TCC 影响，仍可检测拦截事件

## 五、核心模块设计

### 5.1 LogStreamMonitor

**职责**：监听 `syspolicyd` 进程日志，实时捕获 Gatekeeper 拦截事件。

```swift
/// 核心设计
/// - 使用 Process 启动 log stream，predicate 过滤 syspolicyd
/// - 逐行解析输出，正则提取 file:// URL 和 bundle_id
/// - 发现拦截事件 → 发布 DetectionEvent 到 EventDeduplicator
///
/// 关键正则（基于实验 A 真实日志）：
///   "GK Xprotect results:.*file://(.+\.app/)"    → 提取 app 路径
///   "GK evaluateScanResult:.*bundle_id: (.+)\)"   → 提取 bundle_id
///   "Prompt shown"                                → 确认弹窗已出现

class LogStreamMonitor: ObservableObject {
    private var process: Process?
    private var outputPipe: Pipe?
    
    func startMonitoring() { /* 启动 log stream 子进程 */ }
    func stopMonitoring()  { /* 终止子进程 */ }
    
    // 解析单行日志，返回结构化事件
    func parseLogLine(_ line: String) -> DetectionEvent? { ... }
    
    // 子进程崩溃自动重启（看门狗）
    private func setupWatchdog() { ... }
}
```

**健壮性设计**（借鉴 Sentinel 的操作后验证模式）：
- **子进程看门狗**：`Process.terminationHandler` 中自动重启，指数退避（1s → 2s → 4s → 最大 30s）
- **管道缓冲区**：使用 `readabilityHandler` 异步读取，避免阻塞
- **正则分级**：定义 3-4 个不同粒度的正则，从精确到宽泛。精确 Pattern 失败时降级到宽泛 Pattern
- **通道健康探针**：连续 5 分钟无任何输出 → 标记通道为 degraded → Menu Bar 展示黄色警告

> [!WARNING]  
> **已知风险**：`log stream` 输出格式不是 Apple 公开 API，可能随 macOS 版本变化。Channel B (FSEventsWatcher) 作为降级通道。

### 5.2 FSEventsWatcher

**职责**：监控文件系统目录，检测新出现的带 quarantine 属性的 `.app`。

```swift
/// 核心设计
/// - 使用 CoreServices FSEventStream（非 DispatchSource，性能更好）
/// - 启用 kFSEventStreamCreateFlagFileEvents（文件级事件，而非目录级）
/// - 监听 kFSEventStreamEventFlagItemCreated + kFSEventStreamEventFlagItemXattrMod
/// - 监控可配置的目录列表
/// - 检测到 .app 变化 → getxattr() 检查 quarantine → 有 → 发布事件
///
/// 为什么用 FSEventStream 而非 DispatchSource？
/// - FSEventStream 支持目录递归监控
/// - DispatchSource.makeFileSystemObjectSource 只能监控单个文件/目录
/// - 实验已验证 FSEvents 即时检测
///
/// xattr 检查使用 C API（零 shell 依赖）：
/// - getxattr(path, "com.apple.quarantine", nil, 0, 0, XATTR_NOFOLLOW) > 0

class FSEventsWatcher: ObservableObject {
    private var stream: FSEventStreamRef?
    
    // 默认监控目录
    static let defaultDirectories: [URL] = [
        .homeDirectory.appending(path: "Downloads"),
        .homeDirectory.appending(path: "Desktop"),
        URL(fileURLWithPath: "/Applications")
    ]
    
    // FSEventStream 参数
    // flags: kFSEventStreamCreateFlagFileEvents | kFSEventStreamCreateFlagUseCFTypes
    // latency: 1.0 秒（平衡响应速度和系统负载）
    
    func startWatching(directories: [URL]) { ... }
    func stopWatching() { ... }
    
    // 检查单个路径的 quarantine 属性（使用 getxattr C API）
    func hasQuarantine(at path: String) -> Bool {
        return getxattr(path, "com.apple.quarantine", nil, 0, 0, XATTR_NOFOLLOW) > 0
    }
    
    // 快速扫描（首次安装时使用）
    func scanApps(in directories: [URL]) -> [DetectionEvent] { ... }
}
```

**去重策略**：
- FSEvents 单个 `.app` 安装可能触发多次事件（实验观测到 7 次，因 Bundle 包含多个文件）
- 在 FSEventsWatcher 内部做路径级去重（3 秒窗口，与 EventDeduplicator 统一），只上报一次
- copy/解压中间态产生的大量事件均被 3 秒窗口合并

### 5.4 QuarantineRemover

**职责**：移除 quarantine 属性，可选自动打开应用。

```swift
/// 核心设计：
/// - Step 1: 无 sudo 执行 xattr -rd（Process 参数化，禁止 shell 拼接）
/// - Step 2: 验证移除成功（getxattr C API 检查，不盲信退出码）
/// - Step 3: 如失败 → 提示用户「此应用需要管理员权限，请联系我们」+ 一键复制终端命令
/// - Step 4: 可选 → NSWorkspace.shared.open() 自动打开

class QuarantineRemover {
    
    enum FixResult {
        case success
        case failure(Error)
    }
    
    func removeQuarantine(at appPath: URL) async -> FixResult {
        // Step 1: 参数化执行（无 shell 解析，无注入风险）
        let process = Process()
        process.executableURL = URL(fileURLWithPath: "/usr/bin/xattr")
        process.arguments = ["-rd", "com.apple.quarantine", appPath.path]
        try? process.run()
        process.waitUntilExit()
        
        // Step 2: 验证（借鉴 Sentinel 操作后验证模式，用 C API 而非 shell）
        if !hasQuarantine(at: appPath.path) {
            return .success
        }
        
        // Step 3: 失败提示（极少见，如 root-owned app）
        return .failure(RemovalError.needsAdmin(
            message: "此应用需要管理员权限修复，请联系我们",
            command: "sudo xattr -rd com.apple.quarantine '\(appPath.path)'"
        ))
    }
    
    func hasQuarantine(at path: String) -> Bool {
        return getxattr(path, "com.apple.quarantine", nil, 0, 0, XATTR_NOFOLLOW) > 0
    }
    
    func openApp(at path: URL) {
        NSWorkspace.shared.open(path)
    }
}
```

> [!TIP]
> **操作后验证**是从 Sentinel 学到的关键模式。不盲信 `Process` 退出码，用 `getxattr` C API 实际检查隔离属性是否已被移除。

## 八、技术风险与缓解

| # | 风险 | 严重性 | 概率 | 缓解策略 |
|---|------|--------|------|----------|
| R1 | Apple 未来遮蔽 XProtect results 中的 file:// 路径 | 高 | 中 | FSEvents (Channel B) 完全独立于日志，作为降级通道。即使 Channel A 失效，核心功能不受影响 |
| R2 | `log stream` 输出格式随 macOS 版本变化 | 中 | 高 | 正则表达式版本化配置；每次 macOS 大版本更新时回归测试 |
| R3 | FSEvents 误触发（非 quarantine 的文件事件） | 低 | 中 | 先 `xattr -p` 确认再触发通知，误报率为零 |
| R4 | macOS 未来版本移除弹窗机制 | 中 | 低 | GreenLight 不依赖弹窗存在。FSEvents 检测 + xattr 检查的工作流完全独立于弹窗 |
| R5 | `log stream` 子进程占用过多资源 | 中 | 低 | `--predicate` 已严格过滤；监控 CPU/内存，超阈值触发告警 |
| R6 | 部分 syspolicyd 日志被标记 `<private>` | 低 | 已知 | 实验已确认 `file://` URL 和 `bundle_id` 未被遮蔽；`<private>` 字段非关键字段 |

## 附录 C：AppRecord 状态机

```
                      ┌─────────┐
      检测到 ──────►  │ pending │  (🟡 黄灯)
                      └────┬────┘
                           │
              EventDeduplicator 3s 窗口到期
                           │
                      ┌────▼────┐
                      │ blocked │  (🔴 红灯)
                      └────┬────┘
                           │
          ┌────────────────┼────────────────┐
          │ [忽略]         │ [修复]         │ [修复并打开]
          ▼                ▼                ▼
    ┌───────────┐    ┌─────────┐      ┌─────────┐
    │ dismissed │    │ cleared │      │ cleared │
    │ (仍🔴)    │    │ (🟢)    │      │ (🟢)    │
    └─────┬─────┘    └─────────┘      └────┬────┘
          │                                 │
          │                           NSWorkspace.open()
          │
    仍显示在 Popover 🔴 列表中
    不再推送同路径通知（直到文件变化）
    用户可随时在 Popover 手动修复
```

| 状态 | 进入条件 | UI 表现 | 可执行动作 | 退出条件 |
|------|---------|---------|-----------|---------|
| **pending** (🟡) | 检测层发出 DetectionEvent | 不直接展示给用户 | 无（系统内部状态） | 3s 去重窗口到期 → blocked |
| **blocked** (🔴) | EventDeduplicator 输出 GreenLightEvent | 通知推送 + Popover 红灯列表 + Menu Bar 🔴 | 忽略 / 修复 / 修复并打开 | 用户操作 |
| **dismissed** (🔴) | 用户点击 [忽略] | 仍在 Popover 红灯列表（不再推送通知） | 修复 / 修复并打开 | 用户手动修复 → cleared；文件变化 → 重新 blocked |
| **cleared** (🟢) | xattr 移除成功验证 | Popover 绿灯列表 + 绿灯计数 +1 + Menu Bar 🟢 | 无 | app 更新后重新隔离 → blocked（绿灯 ×N 累加） |

**"忽略"语义**：
- 关闭当前通知，但**不是永久白名单**（白名单是 V2 功能）
- 该 app 仍显示在 Popover 🔴 列表中，用户可随时手动修复
- 不再重复推送同一路径的通知（直到该 app 的文件发生变化，如版本更新）
